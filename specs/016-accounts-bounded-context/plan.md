# Implementation Plan: Accounts Bounded Context

**Branch**: `016-accounts-bounded-context` | **Date**: 2026-04-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/016-accounts-bounded-context/spec.md`

## Summary

Build the `accounts/` bounded context within `apps/control-plane/src/platform/`. This covers user self-registration, email verification (SHA-256 hashed token, 24h TTL), configurable signup modes (`open`, `invite_only`, `admin_approval`), invitation-based registration (bypasses verification + approval), admin approval queue and workflow, full account lifecycle state machine (pending_verification ‚Üí active ‚Üí suspended/blocked ‚Üí archived), 8 admin lifecycle actions, default workspace provisioning event on first activation, resend rate limiting via Redis, and Kafka event emission on all lifecycle transitions.

## Implementation Phases

### Phase 1 ‚Äî Setup & Bounded Context Structure
- Create `src/platform/accounts/` package with all module stubs (`__init__.py` + empty files)
- Add accounts-specific settings to `PlatformSettings`: `ACCOUNTS_SIGNUP_MODE`, `ACCOUNTS_EMAIL_VERIFY_TTL_HOURS`, `ACCOUNTS_INVITE_TTL_DAYS`, `ACCOUNTS_RESEND_RATE_LIMIT`
- Alembic migration `003_accounts_tables.py`: 4 tables + all indexes

### Phase 2 ‚Äî US1: Self-Registration and Email Verification
- `models.py`: `User`, `EmailVerification` models + `UserStatus`, `SignupSource` enums
- `schemas.py`: `RegisterRequest` (password strength validator), `VerifyEmailRequest`, `ResendVerificationRequest`, `RegisterResponse`, `VerifyEmailResponse`
- `state_machine.py`: `VALID_TRANSITIONS` + `validate_transition()`
- `exceptions.py`: `InvalidTransitionError`, `RateLimitError`, `InvalidOrExpiredTokenError`
- `repository.py`: `create_user()`, `get_user_by_email()`, `create_email_verification()`, `get_verification_by_token_hash()`, `consume_verification()`
- `email.py`: `send_verification_email()` stub (calls notifications interface or logs in dev)
- `service.py`: `register()`, `verify_email()`, `resend_verification()` ‚Äî including signup mode check, anti-enumeration responses, Redis rate limiting, SHA-256 token hashing
- `router.py`: `POST /register`, `POST /verify-email`, `POST /resend-verification`
- Events: `accounts.user.registered`, `accounts.user.email_verified`, `accounts.user.activated`

### Phase 3 ‚Äî US2: Admin Approval Workflow
- `models.py`: Add `ApprovalRequest` model + `ApprovalDecision` enum
- `schemas.py`: `ApproveUserRequest`, `RejectUserRequest`, `PendingApprovalItem`, `PendingApprovalsResponse`
- `repository.py`: `create_approval_request()`, `get_pending_approvals()` (paginated), `get_approval_request_for_update()` (SELECT FOR UPDATE)
- `service.py`: `approve_user()`, `reject_user()` ‚Äî concurrency-safe via SELECT FOR UPDATE
- `router.py`: `GET /pending-approvals`, `POST /{user_id}/approve`, `POST /{user_id}/reject`
- Events: `accounts.user.approved`, `accounts.user.rejected`

### Phase 4 ‚Äî US3: Invitation-Based Registration
- `models.py`: Add `Invitation` model + `InvitationStatus` enum
- `schemas.py`: `CreateInvitationRequest`, `AcceptInvitationRequest`, `InvitationResponse`, `InvitationDetailsResponse`
- `repository.py`: `create_invitation()`, `get_invitation_by_token_hash()`, `consume_invitation()`, `revoke_invitation()`, `list_invitations_by_inviter()`
- `service.py`: `create_invitation()`, `get_invitation_details()`, `accept_invitation()` ‚Äî bypasses email verification + approval; calls `auth_service.create_user_credential()`
- `router.py`: `GET /invitations/{token}`, `POST /invitations/{token}/accept`, `POST /invitations`, `GET /invitations`, `DELETE /invitations/{id}`
- Events: `accounts.invitation.created`, `accounts.invitation.accepted`, `accounts.invitation.revoked`

### Phase 5 ‚Äî US4: Admin Lifecycle Management
- `schemas.py`: `SuspendUserRequest`, `BlockUserRequest`, `ArchiveUserRequest`, `ReactivateUserRequest`, `UnblockUserRequest`, `ResetPasswordRequest`
- `repository.py`: `update_user_status()`, `get_user_by_id_for_update()` (SELECT FOR UPDATE)
- `service.py`: `suspend_user()`, `reactivate_user()`, `block_user()`, `unblock_user()`, `archive_user()`, `reset_mfa()`, `reset_password()`, `unlock_user()` ‚Äî all call `validate_transition()`, then `auth_service.invalidate_user_sessions()` for status changes that disable login

- Events: all 7 lifecycle event types (suspended, reactivated, blocked, unblocked, archived, mfa_reset, password_reset_initiated)

## Key Decisions (from research.md)

1. **Email verification tokens**: `secrets.token_urlsafe(32)` + SHA-256 hash stored in DB ‚Äî never plaintext
2. **Invitation tokens**: Same pattern as email verification tokens; 7-day TTL
3. **State machine**: Dict-based `VALID_TRANSITIONS` in `state_machine.py` ‚Äî service-layer enforcement
4. **Session invalidation**: In-process call `auth_service.invalidate_user_sessions(user_id)` on suspend/block/archive
5. **Password delegation**: accounts validates strength in Pydantic; auth context hashes via in-process call
6. **Resend rate limiting**: Redis `resend_verify:{user_id}` counter, 3600s TTL, max 3/hour
7. **Concurrent approval**: `SELECT FOR UPDATE` in repository ‚Äî prevents double-approve race
8. **Kafka topic**: `accounts.events` (new topic ‚Äî needs constitution registry update)
9. **Signup mode**: `ACCOUNTS_SIGNUP_MODE` env var ‚Äî not DB-backed config
10. **Anti-enumeration**: Always `202` for register and resend-verification regardless of email existence
11. **No password_hash in accounts**: `User` model owns identity/lifecycle; `auth.UserCredential` owns credentials
