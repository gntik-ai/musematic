# Contract — CI gates for Helm chart changes

UPD-053's "snapshot diff + helm install --dry-run" CI gates land as steps inside the existing `helm-lint` and `e2e` jobs in `.github/workflows/ci.yml` rather than a new workflow file. Per research R7.

## New step in `helm-lint` job

After the existing `Lint charts` step (`.github/workflows/ci.yml:1095-1103`), append:

```yaml
      - name: Render snapshots
        run: |
          mkdir -p /tmp/snapshots
          helm template release deploy/helm/platform -f deploy/helm/platform/values.prod.yaml \
            --kube-version 1.29.0 \
            --api-versions cert-manager.io/v1/Certificate \
            --api-versions cert-manager.io/v1/ClusterIssuer \
            > /tmp/snapshots/prod.rendered.yaml
          helm template release deploy/helm/platform -f deploy/helm/platform/values.dev.yaml \
            --kube-version 1.29.0 \
            --api-versions cert-manager.io/v1/Certificate \
            --api-versions cert-manager.io/v1/ClusterIssuer \
            > /tmp/snapshots/dev.rendered.yaml
      - name: Diff against committed snapshots
        run: |
          diff -u deploy/helm/platform/.snapshots/prod.rendered.yaml /tmp/snapshots/prod.rendered.yaml \
            || (echo "::error::prod snapshot drift — see https://docs.musematic.ai/operations/helm-snapshot/#regenerating to refresh"; exit 1)
          diff -u deploy/helm/platform/.snapshots/dev.rendered.yaml /tmp/snapshots/dev.rendered.yaml \
            || (echo "::error::dev snapshot drift — see https://docs.musematic.ai/operations/helm-snapshot/#regenerating to refresh"; exit 1)
```

The committed snapshots live at:

- `deploy/helm/platform/.snapshots/prod.rendered.yaml`
- `deploy/helm/platform/.snapshots/dev.rendered.yaml`

Both are checked in. They are regenerated by the `Makefile` target `helm-snapshot-update`:

```makefile
helm-snapshot-update:
	helm dependency update deploy/helm/platform
	helm template release deploy/helm/platform -f deploy/helm/platform/values.prod.yaml \
		--kube-version 1.29.0 \
		--api-versions cert-manager.io/v1/Certificate \
		--api-versions cert-manager.io/v1/ClusterIssuer \
		> deploy/helm/platform/.snapshots/prod.rendered.yaml
	helm template release deploy/helm/platform -f deploy/helm/platform/values.dev.yaml \
		--kube-version 1.29.0 \
		--api-versions cert-manager.io/v1/Certificate \
		--api-versions cert-manager.io/v1/ClusterIssuer \
		> deploy/helm/platform/.snapshots/dev.rendered.yaml
```

Snapshot drift is intentional after any template or values change — the developer runs `make helm-snapshot-update`, reviews the diff, commits both files, and the CI gate passes.

## New step in `e2e` job

The `e2e` job at `.github/workflows/ci.yml:447` already provisions a kind cluster via `helm/kind-action@v1`. After the kind cluster is up, append a step:

```yaml
      - name: Helm install --dry-run against kind
        run: |
          helm dependency update deploy/helm/platform
          helm install platform deploy/helm/platform \
            -f deploy/helm/platform/values.dev.yaml \
            --dry-run --debug \
            --kube-version 1.29.0 \
            --create-namespace --namespace platform-dryrun \
          | tee /tmp/helm-dryrun.log
          if grep -E "Error: " /tmp/helm-dryrun.log; then
            echo "::error::helm install --dry-run produced errors"
            exit 1
          fi
```

`--dry-run` does not actually create the resources but does send them to the API server for validation, exercising:
- Schema validation for built-in resource kinds.
- Mutating/validating webhook admission (for any webhooks installed by the kind-cluster setup).
- Detection of duplicate resource names within the chart.
- Detection of empty required fields that pass static `kubeconform` but fail real K8s admission.

If the kind cluster does not have cert-manager CRDs installed, the `Certificate` and `ClusterIssuer` resources rendered by UPD-053 will fail dry-run with "no matches for kind". The fix is one of:

1. Install cert-manager into the kind cluster as a CI prerequisite step (preferred — matches what the `helm install` would actually do in prod).
2. Skip the cert-manager resources via `--api-versions` exclusion in dry-run (rejected — defeats the gate).

The contract chooses option (1): the e2e job adds a prerequisite step that installs cert-manager via `kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.16.0/cert-manager.yaml` before the dry-run.

## Path filter (existing)

The path-filter at `.github/workflows/ci.yml:73` already triggers `helm` jobs on changes under `deploy/helm/**`. UPD-053 inherits this; no path-filter changes needed.

## Local developer workflow (documented)

`docs/operations/helm-snapshot.md` (NEW) walks contributors through:

1. Make the template change.
2. Run `make helm-snapshot-update`.
3. Inspect the diff (`git diff deploy/helm/platform/.snapshots/`); verify intent.
4. Commit both files in the same PR.
5. CI passes the snapshot-diff step; reviewers see the rendered diff in the PR alongside the source change.
