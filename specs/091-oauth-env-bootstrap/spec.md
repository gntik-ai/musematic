## User Scenarios & Testing *(mandatory)*

### User Story 1 - GitOps Deployment with Pre-Configured OAuth (Priority: P1)

A platform operator deploys Musematic to staging via ArgoCD. They want Google and GitHub OAuth configured and working from the first login, with no manual post-install admin actions, no interactive prompts, and no operator-typed secrets at install time. The operator's GitOps repo holds a Helm values file referencing Kubernetes Secrets (or Vault paths) for the OAuth client secrets; the ArgoCD sync triggers a Helm install; the platform pod boots, reads the `PLATFORM_OAUTH_*` env vars, validates them, writes the client secrets to Vault at canonical paths, upserts the OAuth provider rows in the database with `source=env_var`, emits audit-chain entries, and serves the login page with both provider buttons visible.

**Why this priority**: GitOps-friendliness is the canonical contract for v1.3.0 self-hosted deployments per FR-639 + FR-642 + Rule 42. P1 because (a) without env-var bootstrap, every operator must run a manual post-install API call to configure OAuth — the audit-pass roster's value proposition for v1.3.0 hinges on every constitutional rule being achievable on every deployment, including reproducible GitOps; (b) the infrastructure-as-code promotion (dev → staging → production) story is the single biggest enterprise-adoption blocker today; (c) Rule 43 (OAuth client secrets live in Vault) cannot be honored at install time without a bootstrap path that writes to Vault.

**Independent Test**: A fresh Kubernetes cluster (kind-of-3-nodes OR a real Hetzner cluster following UPD-039's FR-608 guide) is provisioned with Vault in `vault.mode=ha` (per UPD-040). The operator's Helm values file sets: `PLATFORM_OAUTH_GOOGLE_ENABLED=true`, `PLATFORM_OAUTH_GOOGLE_CLIENT_ID=...apps.googleusercontent.com`, `PLATFORM_OAUTH_GOOGLE_CLIENT_SECRET_FILE=/etc/secrets/google-client-secret`, `PLATFORM_OAUTH_GOOGLE_REDIRECT_URI=https://app.musematic.ai/auth/oauth/google/callback`, `PLATFORM_OAUTH_GOOGLE_ALLOWED_DOMAINS=company.com`, `PLATFORM_OAUTH_GITHUB_ENABLED=true`, `PLATFORM_OAUTH_GITHUB_CLIENT_ID=Iv1...`, `PLATFORM_OAUTH_GITHUB_CLIENT_SECRET=...` (direct via Kubernetes Secret + envFrom), `PLATFORM_OAUTH_GITHUB_REDIRECT_URI=https://app.musematic.ai/auth/oauth/github/callback`, `PLATFORM_OAUTH_GITHUB_ALLOWED_ORGS=company-org`. After `helm install platform`, verify (a) both providers exist in the database with `source=env_var` (via `psql platform -c "SELECT provider_type, source, enabled FROM oauth_providers;"`); (b) Vault has `secret/data/musematic/staging/oauth/google/client-secret` and `secret/data/musematic/staging/oauth/github/client-secret` populated (via `vault kv get`); (c) the login page at `https://app.musematic.ai/login` renders both OAuth buttons via the existing `OAuthProviderButtons` component; (d) the signup page at `/signup` renders both buttons; (e) a synthetic OAuth flow completes end-to-end for both providers (Google login → callback → user provisioned with `defaultRole=user`; GitHub login → callback → user provisioned); (f) the audit chain has 2 `auth.oauth.provider_bootstrapped` entries with `source=env_var, enabled=true` and NO secret values in any log or audit metadata; (g) the admin tab `/admin/settings?tab=oauth` shows both providers with `source: env_var` badge; (h) zero plaintext-secret regex matches in `kubectl logs platform-control-plane-...`.

**Acceptance Scenarios**:

1. **Given** a fresh cluster with Vault healthy and the operator's Helm values containing `PLATFORM_OAUTH_GOOGLE_*` and `PLATFORM_OAUTH_GITHUB_*` env vars, **When** the operator runs `helm install platform`, **Then** within 5 seconds of pod startup the bootstrap module reads the env vars, validates them, writes secrets to Vault, upserts the provider rows, and emits `auth.oauth.provider_bootstrapped` audit entries — all without operator intervention.
2. **Given** the bootstrap completes, **When** a user navigates to `/login`, **Then** both Google and GitHub OAuth buttons are visible (rendered by the existing `OAuthProviderButtons` component reading from the live `enabled=true` providers).
3. **Given** a user clicks "Sign in with Google", **When** they complete the OAuth round-trip, **Then** they are redirected to the dashboard, provisioned with the `defaultRole`, and an `auth.oauth.sign_in_succeeded` audit entry is emitted.
4. **Given** the bootstrap completes, **When** the super admin opens `/admin/settings?tab=oauth`, **Then** both providers display with the `source: env_var` badge in the new source-badge UI element.
5. **Given** the operator runs `kubectl logs platform-control-plane-...` after 24 hours, **When** they grep for the canonical secret-leak regex set, **Then** zero matches are returned per Rule 31 + Rule 44.

### User Story 2 - Idempotent Reinstall Preserves Manual Adjustments (Priority: P1)

A super admin has logged into the platform after the initial GitOps install (User Story 1) and used the admin UI to add a second allowed domain (`subsidiary.com`) to the Google OAuth provider. The operator later runs `helm upgrade` with unchanged Helm values — UPD-041 must NOT overwrite the admin's manual change. If the operator EXPLICITLY sets `PLATFORM_OAUTH_GOOGLE_FORCE_UPDATE=true`, then UPD-041 MUST overwrite the value AND emit a critical audit entry warning that a manual change was overridden.

**Why this priority**: Idempotency is constitutionally binding (Rule 42 — "OAuth env-var bootstrap is idempotent"). P1 because (a) without idempotency, every Helm upgrade silently destroys super-admin tuning and creates an unsafe operational dynamic; (b) the FORCE_UPDATE escape hatch is needed for genuine drift-recovery scenarios but MUST emit a critical audit entry to make the override traceable; (c) the audit-pass roster requires every constitutional rule to be enforceable in production, and Rule 42 is unenforceable without this story.

**Independent Test**: Bootstrap Google via env vars (User Story 1 setup with `ALLOWED_DOMAINS=company.com`). Super admin logs in, opens `/admin/settings?tab=oauth`, edits the Google provider's allowed-domains list adding `subsidiary.com`; verifies (a) the database row's `source` flips to `manual`; (b) the audit chain has an `auth.oauth.provider_configured` entry with admin principal + diff. Operator re-runs `helm upgrade platform deploy/helm/platform/` (Helm values unchanged). Verifies (c) the database row STILL has both domains; (d) `source` is STILL `manual`; (e) NO new `auth.oauth.provider_bootstrapped` entry was emitted (idempotent skip). Operator changes Helm values to `PLATFORM_OAUTH_GOOGLE_FORCE_UPDATE=true` and re-runs `helm upgrade`. Verifies (f) the database row's allowed-domains REVERTS to `[company.com]` (matches the env var); (g) `source` flips back to `env_var`; (h) a critical-severity audit entry `auth.oauth.config_reseeded` is emitted naming the override AND a structured-log entry at `severity=warning` ("OAuth FORCE_UPDATE overrode manual changes").

**Acceptance Scenarios**:

1. **Given** an env-var-bootstrapped Google provider AND a super-admin manual edit of allowed-domains, **When** the operator re-runs `helm upgrade` with unchanged values, **Then** the manual edit is preserved and no bootstrap event is re-emitted.
2. **Given** the same setup, **When** the operator re-runs `helm upgrade --set oauth.google.forceUpdate=true`, **Then** the env-var values overwrite the manual edit AND a critical audit entry is emitted naming the override.
3. **Given** an existing manual provider (created via admin UI BEFORE any env-var bootstrap), **When** the operator later sets `PLATFORM_OAUTH_GOOGLE_ENABLED=true`, **Then** the bootstrap detects the existing provider and (without `FORCE_UPDATE`) skips with a structured-log entry; with `FORCE_UPDATE=true`, replaces with a critical audit entry.

### User Story 3 - Super Admin Rotates Client Secret (Priority: P1)

The security team rotates the Google OAuth client secret in Google Cloud Console (e.g., per quarterly rotation policy). A super admin opens the admin OAuth tab, clicks "Rotate client secret" on the Google provider, enters the new secret in a write-only input (the current secret is NEVER displayed per Rule 44), submits. UPD-041 writes a new Vault KV v2 version (UPD-040 native versioning), flushes the per-pod cache, and emits an audit entry. Existing OAuth flows in flight continue using the prior version (cached) until cache TTL expires; new flows use the new version. After the configurable dual-credential window (default 30 minutes), the prior version is destroyed via `vault kv metadata delete` (per UPD-040's `RotatableSecretProvider` rewire).

**Why this priority**: Secret rotation is constitutionally binding (Rule 10 — every credential goes through vault; Rule 44 — rotation responses never return the new secret). P1 because (a) without a rotate-secret action, operators must SSH into a control-plane pod and run `vault kv put` manually — error-prone + outside the audit chain; (b) the dual-credential window is the canonical mitigation for in-flight OAuth flows during rotation; (c) Rule 44 is unenforceable in the UI without the dedicated write-only flow.

**Independent Test**: A bootstrapped Google provider with version 1 of the client secret in Vault. Super admin opens `/admin/settings?tab=oauth`, clicks the Google provider's "Rotate client secret" button (NEW UI element added by UPD-041); a confirmation dialog opens with a write-only input field labeled "New client secret" (no display of current secret). Super admin pastes the new secret + clicks Confirm. UPD-041 (a) calls `POST /api/v1/admin/oauth-providers/google/rotate-secret` with the new secret in the request body (HTTPS-only; never logged); (b) the backend writes via `await secret_provider.put("secret/data/musematic/{env}/oauth/google/client-secret", {"value": new_secret})` — UPD-040's `VaultSecretProvider.put` creates a new KV v2 version (v2); (c) the per-pod cache is flushed via `secret_provider.flush_cache(path="secret/data/musematic/{env}/oauth/google/client-secret")`; (d) an audit entry `auth.oauth.secret_rotated` is emitted with `provider=google, old_version=1, new_version=2, admin=...`, NEVER including the secret value; (e) the response to the admin UI is a `204 No Content` (NOT the new secret value per Rule 44); (f) the next OAuth callback for Google reads the new (v2) secret on cache miss; (g) after 30 minutes (the configurable dual-credential window per UPD-040 design D5), v1 is destroyed via `vault kv metadata delete` per UPD-040 task T012.

**Acceptance Scenarios**:

1. **Given** a bootstrapped Google provider with v1 in Vault, **When** the super admin clicks "Rotate client secret" + enters the new secret, **Then** v2 is written, cache is flushed, audit entry emitted, and the response is 204 (no secret value returned).
2. **Given** an in-flight OAuth flow started before rotation, **When** the rotation occurs, **Then** the flow continues using v1 (cached) for the cache TTL window; subsequent flows use v2.
3. **Given** rotation completed, **When** 30 minutes elapse (the dual-credential window), **Then** v1 is destroyed via `vault kv metadata delete` and only v2 remains readable.
4. **Given** any rotation flow, **When** the security engineer reviews the audit chain + structured logs + admin UI response, **Then** the new secret value appears in NONE of them per Rule 31 + Rule 44.

### User Story 4 - Super Admin Manages Group/Team Role Mappings (Priority: P2)

A super admin configures a Google Workspace group (`admins@company.com`) to map to the platform's `admin` role, and a GitHub team (`company-org/security`) to map to the `security_officer` role. The mappings drive auto-provisioning at first login: a new user signing in via Google whose email is in `admins@company.com` is provisioned with the `admin` role; a user signing in via GitHub whose memberships include `company-org/security` is provisioned with `security_officer`. Existing users' roles are NOT auto-reconciled (changes apply only to future logins). The admin UI shows a managed table (add/edit/delete rows) with validation: group format MUST match `[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}` for Google AND `[a-z0-9-]+/[a-z0-9-]+` for GitHub; role MUST exist in the platform's role catalog.

**Why this priority**: Group-to-role mapping is a high-value enterprise feature (FR-643's enumeration) but is P2 because (a) the existing `OAuthProvider.group_role_mapping` JSONB column already supports the data shape — UPD-041's contribution is the UI table, NOT new schema; (b) most P1 acceptance can be reached via direct DB edit by a super admin for a small mapping count; (c) the env-var bootstrap path (User Story 1) already supports `GROUP_ROLE_MAPPINGS` JSON env var, which covers the GitOps use case. The admin-UI table is the polished enterprise experience.

**Independent Test**: A bootstrapped Google provider. Super admin opens `/admin/settings?tab=oauth` → Google provider card → "Role mappings" sub-section. Clicks "Add row"; enters `admins@company.com` + role `admin` (selected from a dropdown listing the platform's roles); clicks Save. Verifies (a) the database row's `group_role_mapping` JSONB column updates to `{"admins@company.com": "admin"}`; (b) the audit chain has an `auth.oauth.role_mapping_updated` entry with diff. A new user `alice@company.com` (member of `admins@company.com`) signs in via Google for the first time. Verifies (c) the user is provisioned with `role=admin` (not `defaultRole=user`). A second user `bob@company.com` (NOT in `admins@company.com`) signs in. Verifies (d) `bob` is provisioned with `role=user` (defaultRole). Super admin edits the mapping (changes `admin` to `super_admin`). Verifies (e) audit entry; (f) `alice`'s existing role is UNCHANGED (no auto-reconciliation per the spec); (g) a NEW user `carol@company.com` who signs in next is provisioned with `super_admin` (the new mapping).

**Acceptance Scenarios**:

1. **Given** a bootstrapped provider, **When** the super admin adds a group-role mapping via the admin UI, **Then** the mapping persists, an audit entry is emitted, and future first-time-OAuth users matching the group receive the mapped role.
2. **Given** a user already provisioned with `role=user`, **When** the super admin adds a mapping that would have given them `admin`, **Then** the user's existing role is NOT changed (no auto-reconciliation); a separate "Reconcile existing users" admin action MAY be added in a future feature.
3. **Given** any role-mapping change, **When** the operator inspects the admin UI, **Then** the table renders the current mappings with edit + delete buttons + a validation indicator showing any malformed entries.

### User Story 5 - Configuration Export and GitOps Promotion (Priority: P2)

A super admin promotes the staging environment's OAuth configuration to production. They run `platform-cli admin oauth export --env=staging --output=staging-oauth.yaml`. The output file is a YAML with provider configs (provider_type, display_name, enabled, client_id, redirect_uri, scopes, domain_restrictions, org_restrictions, group_role_mapping, default_role, require_mfa, source) AND a `client_secret_vault_path` reference (NOT the secret value). The operator inspects the file (it's diff-able with the production config), commits it to the GitOps repo. On the production cluster, where Vault already contains the production secrets at the same canonical paths, the operator runs `platform-cli admin oauth import --input=staging-oauth.yaml --apply`. UPD-041 reads the file, validates each provider config, checks that the referenced Vault path EXISTS in the production Vault (without reading the secret value — just `list_versions`), produces a diff preview, applies on Confirm, and emits audit entries.

**Why this priority**: Export/import enables environment promotion (dev → staging → production) per FR-645. P2 because (a) a one-time manual provider re-creation in production is a viable workaround for the small per-org provider count (1-2 OAuth providers); (b) the canonical-path scheme (`secret/data/musematic/{env}/oauth/{provider}/client-secret`) is already environment-keyed, so secret cross-contamination is prevented by the path structure; (c) MVP can ship without import/export and add it in a follow-up; the brownfield's User Story 5 is a stretch goal.

**Independent Test**: A staging cluster with bootstrapped Google + GitHub providers. Operator runs `platform-cli admin oauth export --env=staging --output=staging-oauth.yaml`; verifies (a) the YAML file contains both providers with all 12 columns + the new 4 columns from spec correction §5; (b) NO `client_secret` field appears (only `client_secret_vault_path`); (c) the file is human-readable + diff-able. Operator inspects + commits to GitOps repo. On a production cluster (with Vault populated with production secrets at the same paths), operator runs `platform-cli admin oauth import --input=staging-oauth.yaml --dry-run`; verifies (d) the dry-run output enumerates the providers to be created/updated + a diff against the current production state; (e) for each provider, the tool calls `await secret_provider.list_versions(client_secret_vault_path)` and confirms ≥ 1 version exists in production Vault. Operator runs `--apply`; verifies (f) the providers are upserted with `source=imported` (NEW source value); (g) audit entries `auth.oauth.config_imported` are emitted.

**Acceptance Scenarios**:

1. **Given** a configured staging environment, **When** the operator runs the export, **Then** the YAML output contains all provider config minus secret values + a Vault path reference per provider.
2. **Given** the production Vault has the corresponding secret paths populated, **When** the operator runs the import with `--dry-run`, **Then** a diff preview is produced AND each Vault path is confirmed populated.
3. **Given** the production Vault is MISSING one of the referenced paths, **When** the operator runs the import, **Then** the import fails with a clear error pointing at the missing Vault path.
4. **Given** the import succeeds, **When** the super admin opens the production admin OAuth tab, **Then** both providers are present with `source=imported` badge.

### Edge Cases

- **`PLATFORM_OAUTH_GOOGLE_ENABLED=true` but `CLIENT_ID` is missing.** Bootstrap module fails fast at startup with a clear error: `"PLATFORM_OAUTH_GOOGLE_CLIENT_ID is required when PLATFORM_OAUTH_GOOGLE_ENABLED=true"`. The error message is logged at `severity=critical` and the platform pod exits non-zero (forcing Helm rollback / ArgoCD failure visibility).
- **Both `CLIENT_SECRET` and `CLIENT_SECRET_FILE` are set.** Bootstrap module fails fast with `"PLATFORM_OAUTH_GOOGLE_CLIENT_SECRET and PLATFORM_OAUTH_GOOGLE_CLIENT_SECRET_FILE are mutually exclusive; set exactly one"`.
- **`ALLOWED_DOMAINS` empty (Google) or `ALLOWED_ORGS` empty (GitHub).** Bootstrap emits a warning `"PLATFORM_OAUTH_GOOGLE_ALLOWED_DOMAINS is empty; any Google account can authenticate"` but does NOT fail (some operators want this — public-internet-facing demo deployments).
- **Invalid JSON in `GROUP_ROLE_MAPPINGS` / `TEAM_ROLE_MAPPINGS`.** Bootstrap fails fast with a parse-error message including the offending position. The existing provider config (if any) is NOT modified.
- **Unknown role in mapping.** Bootstrap validates each role-name against the platform's role catalog (queried at startup); fails with `"Role 'admin_v2' in PLATFORM_OAUTH_GOOGLE_GROUP_ROLE_MAPPINGS does not exist; valid roles: [user, admin, super_admin, security_officer, ...]"`.
- **Redirect URI is non-HTTPS in production.** When `PLATFORM_ENVIRONMENT in ("production", "staging")` AND the `REDIRECT_URI` does NOT start with `https://`, bootstrap fails per Rule 10 + spec edge case unless `ALLOW_INSECURE=true` AND `PLATFORM_ENVIRONMENT=dev`. Mirror of UPD-040's token-auth refusal pattern.
- **Vault unreachable at bootstrap time.** Per Rule 43 (OAuth client secrets live in Vault, never in the database), bootstrap MUST NOT degrade to a database-stored-secret fallback. Fails fast with `"Vault unreachable; cannot bootstrap OAuth providers. Set PLATFORM_VAULT_MODE correctly OR start Vault before retrying"`. The pod exits non-zero.
- **Reseed action when env vars are not present.** Super admin clicks "Reseed from environment" but the running pod has no `PLATFORM_OAUTH_GOOGLE_*` env vars (e.g., post-bootstrap restart with env vars removed). The action returns `400 Bad Request` with `"PLATFORM_OAUTH_GOOGLE_ENABLED is not set in the running pod; cannot reseed"`.
- **Google provider already exists via IBOR sync.** UPD-040 / UPD-024 + IBOR may have written a provider row from an external source. Bootstrap detects `source != null AND source != "env_var"`; emits a warning `"Existing Google provider with source=ibor; env-var bootstrap skipped to avoid conflict"`; does NOT modify the existing row.
- **Concurrent bootstrap attempts from 2+ control-plane pods at startup.** A row-level UPSERT lock (PostgreSQL `ON CONFLICT DO UPDATE WHERE source != 'manual'`) prevents the race; the first pod's bootstrap wins; subsequent pods detect the existing `source=env_var` row and skip per idempotency.
- **Rotation during an in-flight OAuth callback.** UPD-040's per-pod cache (60-second TTL) bridges the dual-credential window; an in-flight callback resolves the v1 secret (cached) until TTL expiry; the next callback resolves v2 (cache miss).
- **`FORCE_UPDATE=true` on an empty database.** Behaves identically to a fresh bootstrap (no manual config exists to overwrite); the audit entry is `auth.oauth.provider_bootstrapped` (NOT `config_reseeded`) since there was no prior manual config.
- **Test-connectivity action on a misconfigured provider.** UPD-041's UI button calls the existing backend endpoint (`router_oauth.py:220`); the response includes a diagnostic message ("invalid_client_id" / "redirect_uri_mismatch" / etc.) which the UI renders inline. NEVER returns the client secret value.
