## User Scenarios & Testing *(mandatory)*

### User Story 1 - Creator Authors a Context Profile for a New Agent (Priority: P1)

A creator is building a customer-support agent and needs it to retrieve from historical tickets + FAQ knowledge base + product documentation, using hybrid retrieval. They navigate to `/agent-management/{fqn}/context-profile` (NEW route — does NOT exist today per spec correction §5). The editor opens with an empty profile if none exists for the agent's current revision. The creator picks 3 sources from a workspace-visibility-filtered picker (memory / knowledge graph / execution history / tool outputs / external APIs), assigns per-source retrieval strategies (semantic via Qdrant, graph via Neo4j, FTS via OpenSearch, hybrid), configures reranking rules, sets context budget (max tokens, max documents, per-source fractions), enables provenance tagging per source. JSON Schema validation runs inline (Monaco-based per spec correction §10 — NOT CodeMirror; schema fetched via NEW `GET /context-engineering/profiles/schema` per spec correction §14). The creator saves; the profile is persisted as version 1 in the NEW `context_engineering_profile_versions` table per spec correction §2 (versioning is greenfield — does NOT exist today). The creator clicks "Test with query" — the NEW `POST /context-engineering/profiles/{id}/preview` endpoint runs a mock retrieval via the NEW `MockLLMProvider` per Constitution Rule 50 (greenfield primitive per spec correction §6) and returns provenance data showing which sources contributed.

**Why this priority**: Per FR-667 + Rule 45, P1 because (a) the existing `context_engineering` BC (16 endpoints verified per spec correction §1) has NO creator-facing UI today; (b) profile authoring is the canonical creator task — without it, creators cannot configure retrieval strategies for their agents; (c) the mock LLM preview per Rule 50 is the canonical contract for non-destructive test-time execution.

**Independent Test**: A creator with a registered agent at FQN `acme.support-agent` navigates to `/agent-management/acme.support-agent/context-profile`. Verifies (a) editor opens with empty profile; (b) source picker lists 5 source types filtered by workspace visibility (per UPD-053 zero-trust visibility); (c) retrieval strategy selector shows 4 options per source; (d) reranking rules editor allows drag-reorder; (e) context budget controls accept max tokens (default 8000) + max documents (default 50) + per-source fractions; (f) provenance tagging toggle per source; (g) JSON Schema validation runs inline with red squiggles on errors; (h) save creates version 1 in the NEW table; (i) "Test with query" mock retrieval returns provenance data within 2 seconds; (j) audit chain entry `creator.context_profile.created` emitted per Rule 9.

**Acceptance Scenarios**:

1. **Given** a creator with a registered agent, **When** they navigate to the context-profile page, **Then** the Monaco-based editor opens with the profile JSON Schema loaded for inline validation.
2. **Given** the source picker is open, **When** the workspace has visibility grants for some sources but not others, **Then** only visible sources render in the picker (per UPD-053 zero-trust filtering).
3. **Given** the creator saves a valid profile, **When** the save completes, **Then** version 1 is persisted in the NEW `context_engineering_profile_versions` table + audit entry emitted.
4. **Given** the creator clicks "Test with query", **When** the preview endpoint runs, **Then** the NEW `MockLLMProvider` returns canned responses + the provenance panel renders sources with scores, classifications, included/excluded flags.
5. **Given** the creator saves an invalid profile (e.g., max_tokens > workspace quota), **When** the JSON Schema validates, **Then** save is blocked with a clear error message.

### User Story 3 - Creator Rolls Back a Profile to a Previous Version (Priority: P2)

A creator changed the retrieval strategy in profile v3 → v4 and evaluation results got worse. They want to revert. They navigate to `/agent-management/{fqn}/context-profile/history` (NEW route). Click "Rollback to v3" — the NEW `POST /context-engineering/profiles/{id}/rollback/{version}` endpoint creates v5 with v3's content (NEVER destructively mutates v3 or v4) per spec correction §13.

**Why this priority**: Per FR-669 + Rule 45, P2 because (a) versioning is GREENFIELD per spec correction §2 (does NOT exist today — must be introduced); (b) without rollback, creators are stuck with regressions; (c) MVP can ship with raw JSON-export of historical versions (manual rollback by re-pasting); the polished UI is P2.

**Independent Test**: A creator's profile has versions v1, v2, v3, v4. Creator navigates to `/agent-management/acme.support-agent/context-profile/history`. Verifies (a) 4 version rows render with timestamp + author + change summary; (b) clicks "Compare v3 vs v4" — side-by-side diff renders with green/red highlights; (c) clicks "Rollback to v3" — confirmation dialog opens; (d) confirms — backend creates v5 with v3's content; v3 + v4 remain unchanged in the table; (e) audit chain entry `creator.context_profile.rolled_back` emitted with `from_version=v4, to_version=v3, new_version=v5`; (f) agent revisions pinned to `version=3` continue using v3 unaffected (revision-version pinning per FR-669).

**Acceptance Scenarios**:

1. **Given** a profile with 4 versions, **When** the creator opens the history tab, **Then** all 4 versions render with timestamp + author + change summary.
2. **Given** the creator clicks "Compare v3 vs v4", **When** the diff renders, **Then** added lines highlight green + removed lines highlight red.
3. **Given** the creator clicks "Rollback to v3" + confirms, **When** the action commits, **Then** v5 is created with v3's content; v3 + v4 are NOT mutated; audit entry emitted.

### User Story 5 - Creator Previews Contract Against Sample Inputs (Priority: P1)

Before attaching a contract to a revision, the creator wants to verify the contract behaves correctly. In the contract editor, they click "Preview". They provide a sample input (or load a saved one). Run preview. The NEW `POST /contracts/{id}/preview` endpoint per spec correction §6 + Constitution Rule 50 simulates execution against the contract using the NEW `MockLLMProvider` (greenfield per Rule 50 — does NOT exist today). The preview output shows: clauses triggered, satisfied, violated, and the resulting action (continue / warn / throttle / escalate / terminate). Violations link to the exact contract clause in the editor. Saved previews accessible from the contract history tab.

### Edge Cases

- **Profile references a source the workspace doesn't have visibility on**: source picker excludes it with a tooltip per UPD-053 zero-trust visibility filtering.
- **Contract references a tool not in the workspace's allowed tools**: validation fails with a clear message; the auto-complete from `GET /exposed-tools` already filters by workspace visibility.
- **Profile schema version mismatch**: per spec correction §13 — the `_schema_version` field on `content_snapshot` JSONB triggers forward-migration via `ContextProfileMigrator` on read; v1 has no migration (only one schema version exists in this feature).
- **Contract preview against malformed sample input**: preview surfaces parse errors without running; the editor highlights the input parse error.
- **Template forked, original later deleted**: forks remain functional (the `forked_from_template_id` FK is `ON DELETE SET NULL`); a one-time notification explains upstream removal.
- **Multi-language contract text**: editor remains English-only (YAML/JSON); descriptions in natural-language fields are not auto-translated; the surrounding wizard UI is i18n'd per FR-620 6-locale support.
- **Mock LLM provider unavailable**: per Constitution Rule 50 — the mock provider MUST be available; if startup detects the mock cannot be initialized (e.g., the canned-responses fixture file is missing), the platform refuses to start (similar to UPD-040's Vault unreachability behavior).
- **Concurrent profile saves from 2 creators**: `ContextEngineeringProfile` updates are protected by row-level UPDATE locks; the second saver receives a 409 Conflict with the latest version number for re-base.
- **Large profile JSONB approaching 1MB**: PostgreSQL JSONB column can hold up to ~1GB but realistically the schema enforces a ~100KB cap on profile content per `_schema_version` validation; oversized profiles fail save with a clear error.
- **Real LLM opt-in for preview**: per User Story 5 acceptance scenario 5 + Rule 50 — the user MUST explicitly confirm + opt-in via a dialog; the audit entry records the opt-in.
