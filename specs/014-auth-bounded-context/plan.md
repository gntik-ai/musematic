# Implementation Plan: Auth Bounded Context

**Branch**: `014-auth-bounded-context` | **Date**: 2026-04-11 | **Spec**: [spec.md](spec.md)

## Summary

Build the `auth/` bounded context within `apps/control-plane/src/platform/`. This includes email/password authentication with Argon2id, RS256 JWT token pair issuance (15min access + 7d refresh), Redis-backed session management, TOTP MFA with encrypted secrets and recovery codes, account lockout/throttling via Redis counters, RBAC engine with 10 predefined roles and workspace-scoped permissions, purpose-bound agent authorization, service account API key management, Kafka event emission for security audit, and 7 REST endpoints.

## Technical Context

**Language/Version**: Python 3.12+  
**Primary Dependencies**: FastAPI 0.115+, Pydantic v2, SQLAlchemy 2.x (async), argon2-cffi 23+ (Argon2id), PyJWT 2.x (RS256), pyotp 2.x (TOTP), redis-py 5.x (async), aiokafka 0.11+, cryptography (Fernet)  
**Storage**: PostgreSQL 16 (models via asyncpg), Redis 7 (sessions + lockout)  
**Testing**: pytest 8.x + pytest-asyncio  
**Target Platform**: Linux server, Kubernetes `platform-control` namespace  
**Project Type**: Bounded context within modular monolith control plane  
**Performance Goals**: Login < 2s, RBAC check < 10ms, token refresh < 500ms  
**Constraints**: Test coverage >= 95%; all async; ruff + mypy --strict  
**Scale/Scope**: 10 RBAC roles, 7 REST endpoints, 6 Kafka event types, 7 PostgreSQL tables

## Implementation Phases

### Phase 1 — Setup & Bounded Context Structure
- Create `src/platform/auth/` package with all module stubs
- Add auth-specific settings to `PlatformSettings` (or extend `AuthSettings`)
- Alembic migration for all 7 tables + permission seed data

### Phase 2 — US1: Email/Password Login with JWT Issuance
- `password.py`: Argon2id hash/verify wrapper
- `tokens.py`: JWT encode/decode with RS256, token pair generation
- `session.py`: Redis session CRUD
- `repository.py`: credential queries
- `service.py`: login flow (verify hash → check lockout → create session → issue JWT)
- `router.py`: POST /login, POST /refresh, POST /logout, POST /logout-all

### Phase 3 — US2: Account Lockout and Throttling
- `lockout.py`: Redis counter increment/check/reset
- Integrate lockout check into login flow
- Auth attempt recording in `auth_attempts` table
- Event emission: `auth.user.locked`

### Phase 4 — US3: MFA (TOTP)
- `mfa.py`: TOTP secret generation (pyotp), Fernet encryption, recovery codes
- MFA enrollment endpoint: POST /mfa/enroll
- MFA confirm endpoint: POST /mfa/confirm
- MFA challenge in login flow
- MFA verify endpoint: POST /mfa/verify

### Phase 5 — US4: RBAC Engine
- `rbac.py`: Permission check against `RolePermission` table
- `UserRole` management (assign/revoke roles)
- In-memory permission cache (loaded on startup, refreshed on change)
- FastAPI dependency for permission checking
- Event emission: `auth.permission.denied`

### Phase 6 — US5: Purpose-Bound Authorization
- `purpose.py`: Purpose-to-action mapping, check function
- Integration with RBAC (called after RBAC passes, for agent identities only)
- Event emission: `auth.permission.denied` with `reason="purpose_violation"`

### Phase 7 — US6: Service Account API Keys
- `ServiceAccountCredential` CRUD
- API key generation (`msk_` + secrets.token_urlsafe), hashing, rotation, revocation
- Auth middleware extension: check `X-API-Key` header before JWT
- Event emission: `auth.apikey.rotated`

### Phase 8 — Polish & Cross-Cutting
- Register auth router in app factory
- Wire auth events into EventTypeRegistry
- All Kafka event emissions with CorrelationContext
- ruff + mypy --strict compliance
- Coverage >= 95%

## Key Decisions (from research.md)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Password hashing | Argon2id via argon2-cffi, OWASP params | Constitution §2.1; PHC format for future migration |
| JWT tokens | RS256 access (15min) + refresh (7d) | Constitution §2.1; stateless access, Redis-checked refresh |
| Session storage | Redis hash per session, user_sessions set | Constitution §III; O(1) lookup, TTL-based cleanup |
| TOTP MFA | pyotp + Fernet-encrypted secrets | Constitution §2.1; encrypted at rest, recovery codes hashed |
| RBAC model | 10 roles, PostgreSQL permission matrix, cached | Predefined roles, seeded via migration, in-memory cache |
| Purpose-bound | Declarative purpose-to-action mapping | Supplements RBAC for agents only; no cross-context DB |
| API keys | msk_ prefix, Argon2id hash, single-use display | One-way hash; prefix for identification without DB lookup |
| Lockout | Redis counter + locked flag with TTL | Atomic increment; auto-expire; no background cleanup |
| Auth events | Kafka canonical EventEnvelope, 6 types | Constitution §IV; enables audit context to consume |
| Auth attempts | PostgreSQL append-only table | ACID audit trail; queryable for lockout verification |
| Password reset | SHA-256 hash, 1-hour TTL | Ephemeral high-entropy tokens; fast hash is sufficient |
