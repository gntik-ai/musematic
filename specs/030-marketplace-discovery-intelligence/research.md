# Research: Marketplace Discovery and Intelligence

**Feature**: 030-marketplace-discovery-intelligence  
**Date**: 2026-04-12  
**Status**: Complete

---

## Decision 1: Bounded Context Ownership

**Decision**: Single bounded context — `marketplace/` — owns all discovery, search, comparison, quality signals, recommendations, contextual discovery, and trending. The `marketplace_intelligence/` module listed in the constitution structure is treated as a sub-package within `marketplace/`.

**Rationale**: All seven user stories share the same root aggregate (the agent listing) and the same write path (agent ratings, quality aggregates). Splitting into two bounded contexts would require cross-boundary service calls for every listing view. Feature 021 (agent registry ingest) already owns the `marketplace-agents` OpenSearch index and `agent_embeddings` Qdrant collection — feature 030 queries those stores without owning them.

**Alternatives considered**: `marketplace/` (search + comparison) + `marketplace_intelligence/` (recommendations + trending) — rejected because the quality aggregate is read by both, requiring shared ownership or a cross-boundary interface; simpler to keep in one context.

## Decision 2: PostgreSQL Tables

**Decision**: Four PostgreSQL tables in the `marketplace/` bounded context:

| Table | Purpose |
|---|---|
| `marketplace_agent_ratings` | User rating + review text (one per user per agent, most recent wins) |
| `marketplace_quality_aggregates` | Pre-computed quality profile per agent — success rate, quality score, self-correction freq, satisfaction avg, cert compliance |
| `marketplace_recommendations` | Pre-computed per-user recommendation list (agent refs + type + score) |
| `marketplace_trending_snapshots` | Trending snapshot computed by scheduled job — agent ref, trending score, growth rate, reason, period |

**Rationale**: MarketplaceListingProjection and TrustSignalProjection are **read projections** assembled at query time from: the `marketplace-agents` OpenSearch index (feature 021, which holds name, FQN, description, capabilities, tags, maturity, cost tier) + `marketplace_quality_aggregates` (this context) + `marketplace_agent_ratings` aggregate. No separate PostgreSQL table for listing projections — that would duplicate registry data.

**Alternatives considered**: Materializing full listing projections in PostgreSQL — rejected because it duplicates data maintained by feature 021's registry. A separate projections table would require maintaining sync on every registry update.

## Decision 3: Search — Reciprocal Rank Fusion (RRF)

**Decision**: Search orchestration queries two sources in parallel then fuses with RRF:
1. **OpenSearch** (`marketplace-agents` index, owned by feature 021): BM25 keyword search with `multi_match` across name, description, capabilities, tags. Workspace visibility pre-filter applied as OpenSearch `filter` clause (FQN pattern allowlist).
2. **Qdrant** (`agent_embeddings` collection, owned by feature 021): Cosine similarity search with embedding of the user query. Workspace visibility applied as Qdrant payload filter (FQN pattern matching).

**RRF formula**: `score(d) = Σ 1 / (k + rank_i(d))` where k=60, rank_i is the 1-based position in each ranked list. Top-N from each source merged; documents appearing in only one source still counted (with implicit rank N+1 for the other source).

**Empty query path**: Skip OpenSearch BM25 + Qdrant semantic; return agents from OpenSearch sorted by `invocation_count_30d` field (popularity within visibility scope) — same visibility filter applies.

**Alternatives considered**: Only BM25 — fails FR-002 (zero keyword overlap). Only semantic — misses precise keyword matches. BM25 + semantic with learned-to-rank — overkill for this feature; RRF is proven to outperform linear combination without training data.

## Decision 4: Embedding for Semantic Search

**Decision**: Query embedding generated by the same embedding API used by feature 021 (`httpx` async call to the model provider embedding endpoint). The query embedding is generated synchronously during the search request. Result: Qdrant `search()` returns agent IDs + scores.

**Rationale**: Agents already have embeddings in Qdrant (feature 021). Reuse the same model for query embedding to ensure semantic compatibility. Query embedding is fast (<100ms) and doesn't need caching for search.

**Alternatives considered**: Pre-embedding common queries — overkill; user queries are too varied for effective pre-computation.

---

## Decision 5: Faceted Filtering

**Decision**: Filtering applied as OpenSearch + Qdrant filter clauses **before** ranking, not as post-filter. Supported facets:

| Facet | OpenSearch filter | Qdrant filter |
|---|---|---|
| Tags | `terms` on `tags` field | `MatchAny` on tags payload |
| Capabilities | `terms` on `capabilities` field | `MatchAny` on capabilities payload |
| Maturity level | `range` on `maturity_level` | `Range` on maturity_level payload |
| Trust tier | `term` on `trust_tier` | `Match` on trust_tier payload |
| Certification status | `term` on `certification_status` | `Match` on certification_status payload |
| Cost tier | `term` on `cost_tier` | `Match` on cost_tier payload |
| Workspace visibility | `terms` on `allowed_workspace_ids` + FQN pattern | FQN payload filter |

**Alternatives considered**: Post-filter — would require over-fetching then filtering; breaks pagination size guarantees.

## Decision 6: Quality Signal Aggregation — Kafka Consumers

**Decision**: `QualitySignalAggregate` is updated by Kafka consumers in the `projection-indexer` runtime profile:

| Kafka Topic | Signal |
|---|---|
| `workflow.runtime` | `step.completed` → increment success counter; `step.failed` → increment failure counter; `step.self_corrected` → increment self-correction counter |
| `evaluation.events` | `evaluation.scored` → update running quality score average |
| `marketplace.rating.created` / `.updated` | Produced by this context → update satisfaction average inline (not via Kafka, to keep aggregate consistent) |
| `trust.events` | `certification.status_changed` → update cert compliance flag |

**Refresh interval**: Quality aggregates updated in near-real-time (within 5 minutes of upstream event — SC-005). Aggregate stored in PostgreSQL `marketplace_quality_aggregates`. No Redis cache (data changes infrequently enough that PostgreSQL read is acceptable).

**"No data yet"**: The aggregate row is created with `has_data = False` until first upstream event arrives. FRs 009-010.

**Alternatives considered**: Polling upstream systems on demand — rejected because SC-005 requires within 5 minutes, and polling would create N×M query load. Computing from raw execution data on query — too slow for marketplace listing pages.

## Decision 7: Recommendations — Three Pipelines

**Decision**: Three recommendation pipelines run as APScheduler background jobs (worker profile), writing to `marketplace_recommendations` PostgreSQL table:

### 7a. Collaborative Filtering (CF)
- **Source**: `workflow.runtime` events → `agent_invocation_log` (in-memory aggregation in worker, periodically flushed to ClickHouse; queried from ClickHouse for CF). Actually — simpler: query the ClickHouse `usage_events` table (feature 020) for the last 30 days: `(user_id, agent_fqn, invocation_count)` pairs.
- **Algorithm**: Item-based CF using cosine similarity of user-agent usage vectors. For each user, find top-K similar users, then recommend agents used by similar users but not by target user. Computed offline, results written to `marketplace_recommendations` with type=`collaborative`.
- **Schedule**: Daily (APScheduler cron, 02:00 UTC).
- **Fallback**: Workspace-popularity ranking when fewer than 5 similar users found.

### 7b. Content-Based
- **Source**: User's recent agent invocations → fetch their capability embeddings from Qdrant → compute centroid → Qdrant `search()` for nearest agent embeddings not yet used by user.
- **Algorithm**: Centroid of last-10-used agent embeddings → kNN search in Qdrant `agent_embeddings` collection.
- **Schedule**: On-demand at query time (fast enough: single Qdrant call). Results cached in `marketplace_recommendations` with type=`content_based`, TTL 6 hours (Redis key `rec:content:{user_id}`).

### 7c. Contextual
- **Source**: Caller provides `context_type` (workflow_step, conversation, fleet_config) + `context_text` in API request.
- **Algorithm**: Embed `context_text` → Qdrant kNN search → filter by workspace visibility → return top-K. Computed on-demand per request.
- **No caching** (context changes with each request).

**Alternatives considered**: Matrix factorization (ALS) — better accuracy but requires PyTorch/Spark dependency; out of scope for this feature. Real-time CF — too slow for request-time computation; offline daily batch is sufficient.

## Decision 8: Trending Score Computation

**Decision**: APScheduler job runs daily at 03:00 UTC. For each agent visible in the platform:
1. Query ClickHouse `usage_events`: `invocations_this_week` (last 7 days), `invocations_last_week` (8-14 days ago).
2. Trending score = `invocations_this_week / max(invocations_last_week, 1)` — growth ratio.
3. Filter: only agents with `invocations_this_week >= 5` qualify (prevents 0→1 showing as "infinite" trending).
4. Top-20 by trending score written to `marketplace_trending_snapshots`.
5. Trending reason: "Nx more invocations this week" where N = round(growth_ratio).
6. Satisfaction delta: compare avg quality score this week vs last week from ClickHouse.

**Alternatives considered**: Real-time trending via Redis sorted sets — more complex; daily refresh is sufficient for SC-008.

## Decision 9: Agent Ratings and Reviews

**Decision**:
- `marketplace_agent_ratings` table: `(user_id, agent_id, score, review_text, created_at, updated_at)`. Unique constraint on `(user_id, agent_id)` — enforced via PostgreSQL unique index + `ON CONFLICT DO UPDATE`.
- **Invocation gate**: Before accepting a rating, query ClickHouse `usage_events` for `(user_id, agent_id)` with `invocation_count > 0`. If none, reject with 403 (FR-023).
- **Aggregate update**: After upsert, re-compute satisfaction average in-process (SELECT AVG from ratings table) and update `marketplace_quality_aggregates.satisfaction_avg`.
- **Creator analytics**: Query ClickHouse `usage_events` for invocation count + trend + failure patterns. `GET /marketplace/analytics/{agent_id}` — only accessible by the agent's namespace owner.
- **Common failure patterns**: Top-3 `error_type` values from ClickHouse `usage_events` filtered by `status=failed` for the agent, last 30 days.

**Alternatives considered**: Separate ratings aggregate table — unnecessary; can compute from `marketplace_agent_ratings` with a GROUP BY.

## Decision 10: Contextual Discovery in Workbenches

**Decision**: Contextual discovery is a thin API layer over the search+recommendation stack:
- `POST /marketplace/contextual-suggestions` — accepts `context_type` + `context_text` + `workspace_id`.
- Calls content-based recommendation pipeline (Decision 7b) with context_text as the "user context".
- Returns top-5 agents with relevance scores.
- Response includes `has_results` flag; if no agents match, returns empty list with `has_results=False` (FR-025).

**No new algorithm needed** — reuses the embedding → Qdrant search pipeline from recommendations.

## Decision 11: Comparison View

**Decision**: Comparison is a pure read operation — no new database queries beyond what's already needed for listings.
- `GET /marketplace/compare?agent_ids=id1,id2,id3` — validates 2-4 IDs, fetches listing projections for each, assembles side-by-side response.
- Difference highlighting: computed in the API response — each attribute includes a `differs` boolean set to `True` if not all compared agents share the same value.
- Attributes: capabilities, maturity_level, trust_tier, certification_status, quality_score_avg, cost_tier, success_rate, user_rating_avg.
- Validation: < 2 → 400 "Minimum 2 agents required"; > 4 → 400 "Maximum 4 agents allowed" (FR-008).

## Decision 13: Runtime Profile Placement

**Decision**:
- **API router**: `api` profile — search, comparison, listings, ratings, creator analytics, contextual suggestions
- **Background workers**: `worker` profile — collaborative filtering job (daily), trending computation (daily), quality aggregate Kafka consumers
- **projection-indexer** profile (if exists): quality signal Kafka consumers could run here instead

**Recommendation**: Quality signal consumers run in `worker` profile alongside other background jobs (same aiokafka consumer group).

---

## Decision 14: Visibility Enforcement

**Decision**: Workspace visibility filtering is enforced at query time in every API endpoint:
1. Retrieve the requesting user's `workspace_id` from the JWT/session.
2. Call `WorkspaceService.get_visibility_config(workspace_id)` in-process → returns allowed FQN patterns list.
3. Apply as filter clause in every OpenSearch query and every Qdrant search.
4. For empty-query browsing and trending: same filter applied.

**Zero-trust**: An agent not in the user's visibility scope never appears in any response. SC-011: 0% leakage.
