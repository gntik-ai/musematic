## Phase 3: User Story 1 — Auto-Triage of Operator-Visible Incidents to On-Call (P1) 🎯 MVP

**Story goal**: Every supported alert-rule firing produces an internal incident AND a corresponding external alert in every enabled integration; deduplicates the same active condition; survives provider unreachability without losing the alert; resolves cleanly; integration credentials never appear outside Vault.

**Independent test**: Configure a single paging-provider integration; trigger each supported alert rule class; verify (a) one incident row per firing, (b) one external-alert row per (incident, integration), (c) provider mock receives the correctly-shaped payload with mapped severity, (d) re-firing the same condition does NOT duplicate, (e) provider outage queues for retry without losing the local incident, (f) resolving the incident closes the external alert.

## Notes

- The `[Story]` tag maps each task to its user story (US1, US2, US3, or US-FE for frontend tasks that span stories) so independent delivery is preserved.
- The `IncidentTriggerInterface` (T016) is the single most important architectural decision in this feature: it keeps incident creation in-process, makes future producers (certifications, security events, chaos detectors) trivial to wire, and lets the analytics hook (T029) stay tiny. Document it loudly in `CLAUDE.md` (T078) so future planners do not reintroduce a Kafka-driven alert path.
- The two Kafka event types (`incident.triggered`, `incident.resolved`), the two REST prefixes (`/api/v1/incidents/*`, `/api/v1/runbooks/*`), and the bounded-context slot (`incident_response/`) are ALREADY declared in the constitution (lines 495, 777–778, 806–807). T009 / T015 / T030 / T043 implement what is already named — do not invent new names.
- Migration `063_incident_response.py` MUST rebase to the current alembic head at merge time (latest at branch cut: `062_cost_governance` from feature 079).
- The brownfield input nominated `analytics/services/alert_rules.py` as the modification target. That file does not exist; T029 puts the hook in the existing `analytics/service.py` instead. The deviation is explicit in `plan.md` and re-flagged in T029's description and T078's `CLAUDE.md` update so future agents do not "fix" the missing file.
- Provider-side **two-way state sync** (provider-side ack → platform incident state update) is out of scope for v1 per the spec; the integration UI documents this limitation (T068 `IntegrationConfigForm` shows a help text below the integration row).
- Post-mortem **timeline auto-summarization** via LLM is explicitly out of scope (spec § Out of Scope); rule 50 (mock LLM for previews) is therefore N/A here.
- The `condition_fingerprint` (used for dedup) must be deterministic and stable across upstream callers. The recommended derivation is `sha256(alert_rule_class + ":" + scope_identifier + ":" + scope_value)` where `scope_value` is the workspace_id / fleet_id / agent_id the rule applies to. Document this in `contracts/incident-trigger-interface.md` so future producers fingerprint identically.
