# Tasks: Multi-Region High-Availability Deployment

**Feature**: 081-multi-region-ha
**Branch**: `081-multi-region-ha`
**Plan**: [plan.md](./plan.md) | **Spec**: [spec.md](./spec.md) | **Planning Input**: [planning-input.md](./planning-input.md)

User stories (from spec.md):
- **US1 (P1)** — Active-passive disaster-recovery posture with observable replication (foundational; everything else assumes a trustworthy standby)
- **US2 (P2)** — Documented and auditable failover procedure (failover plans + rehearsal + production execution)
- **US3 (P3)** — Maintenance mode that drains cleanly and speaks plainly (window scheduling + middleware gate + announcement)
- **US4 (P4)** — Zero-downtime platform upgrades through documented patterns (operator runbook + dashboard linkage + agent runtime versioning surface)
- **US5 (P5)** — Capacity planning signals that reach the operator before saturation (composes existing forecasts + usage rollups; never re-implements)

Each user story is independently testable per spec.md.

## Phase 3: User Story 1 — Active-Passive Disaster-Recovery Posture (P1) 🎯 MVP

**Story goal**: A passive secondary is declared, every required data store category is replicating, lag is observable per (component, source, target) tuple on the operator dashboard, RPO threshold breaches fire incidents that auto-clear when lag returns below threshold, paused replication is distinguishable from outage, and missing replication paths are surfaced as explicit gaps.

**Independent test**: Stand up a secondary against a non-production primary; declare RPO target; confirm the dashboard shows lag for all 7 components; pause one component's replication and confirm an alert fires within the platform's stated alert-delivery latency; resume and confirm the alert auto-clears.

## Notes

- The `[Story]` tag maps each task to its user story (US1, US2, US3, US4, US5, or US-FE for frontend tasks that span stories) so independent delivery is preserved.
- The **two architectural inversions** worth flagging loudly to future planners are: (a) the **maintenance gate fails OPEN on Redis miss** (T015, T048, T090) — a deliberate departure from rule 41's fail-closed posture because this gate is cooperative infrastructure protection, not authentication; (b) the **maintenance gate is registered ABOVE auth** in the middleware stack (T016) so unauthenticated callers see the maintenance message rather than chasing a 401-then-503. T090 ensures CLAUDE.md captures both so the next planner doesn't try to "fix" them.
- The **partial UNIQUE index on `region_configs (region_role) WHERE region_role='primary' AND enabled=true`** (T006) is the database-level final guard against active-active misconfiguration — the application-level check in T021 is the friendly error path; the index is the safety net.
- The **fingerprint contract** for incidents fired from this BC follows feature 080's convention: `sha256(component + ":" + source_region + ":" + target_region)` for replication breaches and `sha256("capacity:" + resource_class + ":" + scope_id)` for capacity saturation. This keeps dedup deterministic across producers (T030, T067).
- The 5 Kafka event types (`region.replication.lag`, `region.failover.initiated`, `region.failover.completed`, `maintenance.mode.enabled`, `maintenance.mode.disabled`), the 2 REST prefixes (`/api/v1/regions/*`, `/api/v1/maintenance/*`), the 2 feature flags (`FEATURE_MULTI_REGION`, `FEATURE_MAINTENANCE_MODE`), and the bounded-context slot (`multi_region_ops/`) are ALREADY declared in the constitution (lines 492, 768–772, 798–799, 888–889). T010 / T014 / T003 implement what is already named — do not invent new names.
- Migration `064_multi_region_ops.py` MUST rebase to the current alembic head at merge time (latest at branch cut: `063_incident_response` from feature 080).
- **Capacity NEVER re-implements forecasting** (T066, T068 — explicit absence guarantee). The service composes feature 079's `get_latest_forecast` and feature 020's usage rollups. Re-implementing would create two divergent paths — the spec forbids it (FR-482.6).
- The 2 operator runbooks (`failover.md`, `zero-downtime-upgrade.md`) plus the active-active considerations doc are first-class deliverables, not afterthoughts. FR-480.5 explicitly requires the upgrade procedure reachable from the operator dashboard at the moment of an upgrade — T062 wires the link, T059 authors the content.
- The **planning input nominated 4 SP / 2 days** for this feature; the plan flagged this as materially understated (5 PG tables, 7 probes, hot-path middleware, 3 APScheduler jobs, 5 Kafka schemas, 2 REST router groups, 1 Helm overlay, 7 replication-job templates, 1 Grafana dashboard, 2 runbooks + 1 active-active doc, 3 frontend panels + 14 components, full unit/integration/E2E coverage to ≥ 95%). Recommend descoping into **Wave 9A** (Phases 1–3 + Phase 9 essentials — the MVP that delivers DR observability) and **Wave 9B** (Phases 4–8) if the 2-day budget is firm.
