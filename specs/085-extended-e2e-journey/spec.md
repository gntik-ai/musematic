**Feature Branch**: `085-extended-e2e-journey`
**Created**: 2026-04-27
**Status**: Draft
**Input**: User description: "Unify the observability install into a single `deploy/helm/observability/` umbrella chart that ships every component (Prometheus, Grafana, Jaeger, Loki, Promtail, Alertmanager, OTEL Collector), every dashboard (7 baseline + 14 audit-pass = 21 total), every alert rule, and every Grafana data source — installable in one command — and extend the user-journey E2E suite with 8 new journeys (Privacy Officer, Security Officer, Finance Owner, SRE, Model Steward, Accessibility User, Compliance Auditor, Dashboard Consumer), 6 new bounded-context suites for the audit-pass bounded contexts, 6 new chaos scenarios, and assertion helpers that run against a real Loki + Prometheus + Jaeger stack — no mocks of the observability backends."

> **Constitutional anchor:** This feature IS the constitutionally-named **UPD-035** ("Unified observability Helm bundle + extended journey coverage") declared in Constitution § "Observability Extension (UPD-034 and UPD-035)" (lines 502-516) and named in the audit-pass roster at Constitution line 476. > - **Feature 047** (existing observability stack — Prometheus, Grafana, Jaeger, Alertmanager, OTEL Collector, 7 baseline dashboards) — UPD-035 wraps these into the umbrella chart unchanged.

> - **Feature 045** (installer-operations CLI) — UPD-035 adds the `platform-cli observability install|upgrade|uninstall|status` subcommand on top of the existing CLI surface.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Operator Installs the Complete Observability Stack in One Command (Priority: P1)

A platform operator preparing a fresh deployment MUST be able to install the complete observability stack — Prometheus, Grafana, Jaeger, Loki, Promtail, Alertmanager, OTEL Collector, all data sources, all 21 dashboards, all alert rules — with **one Helm command** (or one CLI subcommand wrapping it). The install MUST be reproducible: the same chart + values produces the same outcome on a kind laptop cluster, on a small managed cluster (GKE/EKS/AKS), and on an on-prem k3s cluster. After install, every component MUST self-report healthy (Prometheus scraping, Grafana datasources reachable, Jaeger accepting spans, Loki accepting log pushes) and the operator CLI's `status` subcommand MUST return ✓ for every component.

**Why this priority**: Without single-command install, every fresh deployment is a multi-step human ritual: install Prometheus, install Grafana, configure datasources, install Loki, install Promtail, import each dashboard, wire each alert. That ritual is the most common cause of "missing dashboard" / "no logs in Grafana" / "alert rule didn't load" production support cases. P1 because (a) every customer deployment hits this path, (b) the chart is the contract that journey tests run against — without it, FR-530 (real-stack E2E) cannot exist.

**Independent Test**: On a fresh kind cluster, run `helm upgrade --install observability ./deploy/helm/observability/ --namespace platform-observability --create-namespace --values ./deploy/helm/observability/values-standard.yaml` and wait for `--wait` to return. Verify (a) every named component pod is `Running` + `Ready`; (b) Grafana's `/api/health` returns 200; (c) Grafana's datasource list contains Prometheus, Loki, and Jaeger and each `/api/datasources/N/health` returns success; (d) Grafana's dashboard list contains all 21 dashboard UIDs; (e) Prometheus's `/api/v1/rules` contains every alert rule shipped by the chart; (f) the Loki ruler's `/loki/api/v1/rules` contains every Loki alert rule from feature 084; (g) `platform-cli observability status` returns ✓ for every component. Run `helm uninstall observability -n platform-observability` and verify zero residual CRDs, PVCs, webhooks, or ConfigMaps remain in the namespace.

**Acceptance Scenarios**:

1. **Given** a fresh Kubernetes cluster with no observability components installed, **When** the operator runs the umbrella `helm upgrade --install observability …` command, **Then** within the documented preset capacity envelope every observability component is `Running + Ready`, every Grafana datasource is healthy, every dashboard is loaded, every alert rule is loaded, and `platform-cli observability status` returns ✓.
2. **Given** the umbrella chart is installed, **When** the operator runs `helm upgrade observability …` with a values change (e.g., higher retention), **Then** the upgrade applies cleanly with no data loss for in-tier (hot) data and the new value is reflected in the running components.
3. **Given** the umbrella chart is installed, **When** the operator runs `helm uninstall observability -n platform-observability`, **Then** every component is removed, the namespace is deletable, and no orphaned CRDs / PVCs labelled by this chart / webhooks / dashboards remain.
4. **Given** the operator selects the `minimal` preset on a 16 GB kind laptop, **When** the chart installs, **Then** the entire observability stack fits inside the documented dev-cluster resource budget (memory + CPU) and remains responsive.
5. **Given** the operator selects the `enterprise` preset, **When** the chart installs, **Then** Prometheus runs in HA mode (with the upstream community chart's HA topology — typically Thanos sidecars or paired replicas) and Loki runs in distributed mode with S3 chunk storage; each preset's exact topology is documented in the chart README.
6. **Given** the umbrella chart's values do not specify a Loki S3 bucket, **When** the operator runs `helm install` with the `standard` or `enterprise` preset, **Then** the install fails fast at template-render time with a clear error indicating that the bucket name is required (no silent fallback to in-cluster disk).
7. **Given** the operator chooses to install on a managed cluster behind an Ingress, **When** the chart's `ingress.enabled=true` is set, **Then** Grafana is reachable via the configured hostname with TLS terminated at the Ingress; the ingress configuration values are documented and tested on at least one managed-cluster path.
8. **Given** Grafana is integrated with platform SSO via the OAuth plugin, **When** an authorised platform user opens Grafana, **Then** the user is signed in via SSO and inherits the constitutionally-defined operator-RBAC scope; an unauthorised user is refused at the SSO step (no parallel auth path inside Grafana).

### User Story 2 - SRE Debugs an Outage End-to-End via the Observability Stack (Priority: P1)

An SRE responding to an alert MUST be able to start at the alert notification, follow a single linear navigation through the unified Grafana shell — alert → dashboard panel → Loki log → Jaeger trace → Prometheus metric → resolution — without leaving Grafana, without manually re-typing queries, and without copy-pasting IDs between tools. The Dashboard Consumer / Log-Metric-Trace Correlation journey (J17) MUST execute this exact path against a real cluster with a real (synthetic) outage triggered, and assert success at every hop.

**Why this priority**: This is the canonical observability ROI scenario — the reason any of the three backends exist. P1 because (a) outage response is on the platform's measured-availability hot path; (b) feature 084's structured-logging contract and feature 047's metric/trace contracts are *only* worth their weight if the navigation-between-tools experience works without friction; (c) the journey is the executable contract that the umbrella chart's data-source provisioning + derived-field configuration + dashboard panels are correctly composed.

**Independent Test**: On a kind cluster with the umbrella chart installed and the platform Helm chart installed and seed data loaded, trigger a controlled synthetic failure (e.g., a seeded agent fails an execution with a known-error log line; an upstream-API latency spike emits a known metric pattern). Verify (a) the failing log line appears in Loki within 15 seconds with all required structured fields; (b) the `HighErrorLogRate` Loki alert (from feature 084) fires and propagates through Alertmanager to the configured notification channel; (c) the Cross-Service Error Overview dashboard (D12) shows the failing service inside its panel; (d) clicking a Loki log entry's `trace_id` derived field opens the corresponding Jaeger trace; (e) the trace contains spans across at least 3 services with correct parent/child relationships; (f) the Prometheus metric for the same service shows a correlated spike at the same timestamp; (g) after the synthetic failure is resolved, the alert closes within one Loki ruler evaluation cycle.

**Acceptance Scenarios**:

1. **Given** a synthetic error condition active on the test cluster, **When** the J17 journey executes, **Then** all 7 hop assertions above pass and the journey produces a narrative HTML report (per FR-529) that includes a Grafana dashboard snapshot taken at the alert-fired moment.
2. **Given** the J17 journey is run repeatedly, **When** the journey runs in parallel with other journeys, **Then** the journey's assertions remain deterministic (no flakiness from shared cluster state) — the synthetic-failure mechanism is scoped via correlation_id / trace_id labels so concurrent journeys do not contaminate each other.
3. **Given** the umbrella chart is correctly configured, **When** any operator opens Grafana on a fresh install, **Then** the data-source list shows Prometheus, Loki, and Jaeger; clicking Loki's "Explore" view shows derived-field links for `trace_id` (→ Jaeger) and `correlation_id` (→ filtered Loki query) per FR-515.
4. **Given** the synthetic outage is resolved, **When** the Loki ruler re-evaluates, **Then** the `HighErrorLogRate` alert closes within one evaluation cycle and the close-event propagates to the notification channel.
5. **Given** the journey runs on a kind cluster with the `e2e` preset, **When** the umbrella chart's resource limits are applied, **Then** the journey completes within its declared time budget without OOM-killing any observability component.

### User Story 3 - Privacy Officer Completes a Right-To-Be-Forgotten Cycle End-to-End (Priority: P1)

A Privacy Officer responding to a data subject erasure request MUST be able to validate identity, trigger the cascade erasure, and verify completion across every data store — without piecing together evidence from per-table queries. The Privacy Officer journey (J10 in the brownfield numbering — final number subject to feature-072 reconciliation) MUST exercise this flow end-to-end against the real `privacy_compliance` bounded context (feature 076) and assert that cascade deletion propagates across PostgreSQL, Qdrant, Neo4j, ClickHouse, OpenSearch, and S3, that a tombstone record is created with cryptographic proof, that the audit chain (feature 074/UPD-024) records the operation, and that the subject is notified via the configured channel (feature 077/UPD-028).

**Why this priority**: Right-to-be-forgotten is the canonical regulatory hot-path test. Without an end-to-end journey, the per-store unit tests can each pass while the cascade fails (e.g., the OpenSearch deletion is silently skipped, the audit chain entry is written but doesn't cover the right scope, the notification fires before deletion completes). P1 because regulatory cycles bind organisational risk; the journey IS the regulatory-compliance evidence.

**Independent Test**: On a seeded test cluster with a known test user, submit a DSR via `POST /api/v1/privacy/dsr` (the public self-service endpoint), have the Privacy Officer fixture approve it, wait for cascade, and verify (a) PostgreSQL rows for that user are tombstoned per the feature-076 contract; (b) Qdrant vectors associated with that user are removed; (c) Neo4j nodes for that user are detached and tombstoned; (d) ClickHouse rows for that user are hard-deleted (per the regulatory hard-delete requirement, not tombstoned); (e) OpenSearch documents are deleted; (f) S3 objects under the user's namespace are purged; (g) the tombstone record exists in PostgreSQL with a cryptographic hash covering the deletion scope; (h) the audit chain entry exists, links to the tombstone, and the chain integrity is verifiable end-to-end after the operation; (i) the subject receives a notification via the configured delivery method; (j) the DSR's status is `completed` with timestamp and duration metric.

**Acceptance Scenarios**:

1. **Given** a seeded test user with data spread across all six stores, **When** the J10 Privacy Officer journey executes the DSR cycle, **Then** all 10 cascade-completion checks above pass with at least 20 distinct assertion points.
2. **Given** a DSR is in flight, **When** the Privacy Officer cancels before completion, **Then** the cascade is rolled back per the feature-076 contract and no tombstone record is created (negative-test variant).
3. **Given** a DSR is completed, **When** an auditor later runs `POST /api/v1/audit/verify` for the period containing the DSR, **Then** the chain integrity check returns ✓ — the J10 journey asserts this verification step explicitly.
4. **Given** one of the six store deletions partially fails (e.g., Qdrant returns 503), **When** the cascade scanner runs the next reconciliation cycle, **Then** the partial-failure path is exercised and the journey asserts that the DSR is NOT marked `completed` and the tombstone is NOT created — `failed` is the correct outcome (negative-test variant).

### User Story 4 - Security Officer Completes a Compliance Cycle End-to-End (Priority: P1)

A Security Officer running a quarterly compliance cycle MUST be able to publish an SBOM, ingest a vulnerability scan result, schedule a secret rotation with a dual-credential window, issue a JIT credential, verify audit-chain integrity for the period, and export a signed audit log — all through the platform's APIs and dashboards, with each step producing audit evidence that the next step references. The Security Officer journey (J11) MUST exercise this end-to-end flow against the real `security_compliance` (UPD-024), `audit/` (extended by UPD-024), and `notifications/` (extended by UPD-028) bounded contexts.

**Why this priority**: Same regulatory-deadline binding logic as US3. Security reviews are the second of the two canonical periodic audits. P1 because the platform's operational integrity claim depends on the audit chain being verifiable on demand and the JIT credential lifecycle being tracked end-to-end.

**Independent Test**: Publish an SBOM (in both SPDX and CycloneDX formats), ingest a vulnerability scan with a known critical CVE, schedule a database-credential rotation with a 7-day dual-credential window, issue a JIT credential to the on-call fixture for a specific operation, verify the audit chain integrity for the period, and export a signed audit log. Verify (a) the SBOM is retrievable in both formats; (b) the scan result is visible with severity breakdown; (c) the critical CVE triggers an incident-integration ticket creation (per feature 080/UPD-031); (d) the rotation is scheduled and old credentials remain valid through the dual-credential window; (e) the JIT credential's use is audit-logged with scope + TTL; (f) the audit chain integrity check returns ✓ for the test period; (g) the signed export's external-signature verification succeeds using the published public key.

**Acceptance Scenarios**:

1. **Given** a seeded security_compliance state, **When** the J11 Security Officer journey executes, **Then** all 7 cycle assertions above pass with at least 20 distinct assertion points.
2. **Given** a critical CVE is found in a development dependency, **When** the journey ingests the scan result, **Then** the CVE is recorded but does NOT block the platform release (constitution Integration Constraint), and the journey asserts that distinction.
3. **Given** the JIT credential is issued and used, **When** the credential's TTL expires, **Then** the next operation attempting to use it is refused with a clear authorisation error and the audit log records the refusal — the journey asserts both.
4. **Given** the signed audit export is generated, **When** the external auditor (test fixture) verifies the signature with the published public key, **Then** the verification succeeds.

### User Story 5 - Finance Owner Completes a Budget and Chargeback Cycle End-to-End (Priority: P1)

A Finance Owner managing workspace budgets MUST be able to configure a monthly budget with soft thresholds and a hard cap, observe alerts firing at 50% and 80%, see the hard cap block new executions, apply an admin override to unblock, observe a cost anomaly (sudden model-cost spike), and export a chargeback report — all through the platform's APIs and dashboards. The Finance Owner journey (J12) MUST exercise this flow against the real `cost_governance/` (UPD-027) bounded context with notifications routed to a real channel via feature 077 (UPD-028) and analytics rollups visible in feature 020's analytics surfaces.

**Why this priority**: Cost-governance is the audit-pass feature with the largest blast-radius for money-direct outcomes. P1 because (a) budget over-runs are an operational and contractual risk; (b) feature 079's contract — "soft thresholds notify, hard cap blocks, admin override unblocks with audit, anomaly detection raises an event" — is testable only end-to-end, not unit; (c) the chargeback report is the artifact finance teams hand to billing.

**Independent Test**: Configure a workspace budget via `POST /api/v1/costs/budgets`, trigger executions that cross 50% and 80% thresholds (verify alerts), trigger executions that cross the hard cap (verify block), apply an admin override (verify unblock + audit), trigger a 10× model-cost spike via the Mock LLM Provider's cost-injection knob, verify the anomaly is detected and listed, export the chargeback report with correct cost breakdown, and check the end-of-period forecast with confidence interval. Verify (a) all 4 alert/block/override transitions fire; (b) the anomaly feed shows the spike with the correct factor; (c) the chargeback report's totals match the analytics rollups; (d) the forecast surfaces a documented end-of-period number with a confidence interval.

**Acceptance Scenarios**:

1. **Given** a seeded workspace with a configured budget, **When** the J12 Finance Owner journey executes, **Then** all 18 assertion points (4 transitions + anomaly + chargeback + forecast + cost-attribution checks) pass.
2. **Given** the budget is configured with a hard cap, **When** an execution would cross the cap, **Then** the execution is refused with a clear cost-cap-blocked error message — the journey asserts the error message text.
3. **Given** an admin applies an override, **When** the override is logged, **Then** the audit chain entry references the original-budget snapshot and the override-grant scope.
4. **Given** the period ends, **When** the forecast is computed, **Then** the forecast appears in the cost dashboard with a documented confidence interval and is queryable via the API.

### User Story 6 - SRE Completes a Quarterly Failover Test End-to-End (Priority: P1)

An SRE running a quarterly failover test MUST be able to schedule a maintenance window, drain in-flight executions gracefully, verify replication lag is below the RPO target across every data store, execute the failover to the secondary region, verify new executions route to the secondary, execute failback, and reconcile data with zero divergence. The SRE / Multi-Region journey (J13) MUST exercise this flow against the real `multi_region_ops/` (UPD-025) bounded context including the maintenance-gate middleware (feature 081), the replication monitor, and the failover orchestrator.

**Why this priority**: Disaster-recovery rehearsals are the canonical periodic operational test. P1 because (a) DR plans that are not executed regularly do not work when needed; (b) the failover orchestrator's contract is verifiable only end-to-end (per-store replication lag is unit-testable, but the orchestrated drain → failover → route → failback sequence is not); (c) the journey is the auditable evidence that DR works.

**Independent Test**: On a multi-region kind cluster (or a single cluster with simulated regions per the feature-081 dev-mode contract), schedule a maintenance window via the maintenance-gate API, verify in-flight executions complete and new executions return the maintenance-mode error, execute the failover, verify new executions succeed against the secondary, execute failback, and run reconciliation queries verifying zero data divergence between primary and secondary. Verify (a) the maintenance window is visible on the operator dashboard; (b) replication lag is below RPO across all stores prior to failover; (c) failover transitions DNS/routing to secondary; (d) new executions succeed against secondary; (e) failback restores primary; (f) reconciliation confirms zero divergence; (g) RPO/RTO metrics for the event are recorded.

**Acceptance Scenarios**:

1. **Given** a seeded multi-region setup with replication healthy, **When** the J13 SRE journey executes, **Then** all 7 stages pass with at least 18 distinct assertion points.
2. **Given** the secondary region is not ready (capacity check fails), **When** the operator attempts failover, **Then** the operation is refused with a clear capacity-error message — the journey asserts this negative path.
3. **Given** failover is executed, **When** the maintenance window auto-closes, **Then** the gate middleware's fail-OPEN-on-Redis-miss behaviour (feature 081) is honoured: stuck Redis state does not block the cluster — the journey asserts this by toggling the Redis lock state mid-test.

### User Story 7 - Model Steward Completes a Catalog and Fallback Cycle End-to-End (Priority: P1)

A Model Steward governing the approved-model catalog MUST be able to approve a new model entry with a complete model card, deprecate an older entry with a grace period, configure a fallback policy for a critical agent (primary + two fallbacks with quality-tier constraints), verify the fallback triggers correctly when the primary returns a rate-limit error, and verify cost attribution reflects the fallback provider's price. The Model Steward journey (J14) MUST exercise this flow against the real `model_catalog/` (UPD-026) bounded context, with the Mock LLM Provider injecting the synthetic 429 response that triggers fallback.

**Why this priority**: Same logic as US5 — this is the executable contract for feature 075's model-fallback policy and grandfathering rules. P1 because model availability outages are common in practice and the platform's promised graceful-degradation behaviour is unverifiable without the journey.

**Independent Test**: Approve a model entry via `POST /api/v1/model-catalog/entries` with a full model card, deprecate an existing entry with a grace-period date, configure a fallback policy on a seeded agent (primary + 2 fallbacks, quality-tier constraints), have the agent execute, then inject a synthetic 429 from the primary via the Mock LLM Provider, verify the agent uses the tier-2 fallback, verify the fallback event is logged + visible in the Model Catalog dashboard (D17), and verify the cost attribution record shows the fallback provider's price. Verify (a) the model card is complete + retrievable; (b) deprecation grace-period is honoured (in-flight executions on the deprecated model continue but new ones cannot bind to it); (c) fallback policy is saved + retrievable; (d) the synthetic 429 triggers the documented fallback behaviour; (e) the fallback event is queryable; (f) cost attribution reflects the fallback price; (g) the audit chain records the model-deprecation step.

**Acceptance Scenarios**:

1. **Given** a seeded model catalog and agent, **When** the J14 Model Steward journey executes, **Then** all 7 stages pass with at least 18 distinct assertion points.
2. **Given** all configured fallbacks fail (cascade exhaustion), **When** the agent's last fallback returns a non-retriable error, **Then** the agent's execution fails with a clear "model unavailable, all fallbacks exhausted" error — the journey asserts this negative path (the test case overlaps with the new chaos scenario `test_model_provider_total_outage.py`).
3. **Given** an agent grandfathered on a deprecated model, **When** the deprecation grace-period elapses, **Then** the agent's next execution must rebind or fail — the journey asserts the rebind/fail dichotomy.

### User Story 8 - Accessibility User Completes a Full Workflow Keyboard-Only with Zero AA Violations (Priority: P1)

A user operating only with keyboard + screen reader MUST be able to log in, navigate the marketplace, start a conversation with an agent, send a message, observe execution progress via ARIA live region announcements, review the reasoning trace, and log out — all without using a mouse, with every status change announced, and with axe-core reporting **zero AA violations** on every visited page. The Accessibility User journey (J15) MUST exercise this flow against the real frontend (apps/web) with keyboard-only Playwright automation and axe-core scanning at every page transition.

**Why this priority**: Constitution rule 28 declares that **axe-core in CI MUST fail the build on any WCAG 2.1 AA violation**. P1 because (a) accessibility is the audit-pass feature with the most legal risk; (b) keyboard-only flows are the canonical disability-affordance test; (c) feature 083 (UPD-030) ships the `localization/` BC and the WCAG-AA implementation, and J15 is its executable contract.

**Independent Test**: Run a Playwright-driven keyboard-only walkthrough: Tab/Shift+Tab through the login form, submit, navigate the marketplace via keyboard, open an agent's detail page, start a conversation, send a message via keyboard, observe ARIA live region announcements during execution, navigate to the reasoning trace, log out. At every page transition run axe-core with the AA ruleset (the same ruleset the CI gate uses) and assert zero violations. Verify (a) every interactive element is reachable via Tab/Shift+Tab/arrows; (b) every status change is announced via ARIA live region; (c) color-contrast ratios are at least 4.5:1 (normal) / 3:1 (large) on every visited page; (d) focus indicators are visible on every interactive element; (e) the screen-reader accessibility tree contains correct landmarks/headings/labels; (f) zero AA violations on any visited page; (g) the command palette (Cmd/Ctrl+K) is fully usable via keyboard.

**Acceptance Scenarios**:

1. **Given** the seeded test cluster with the frontend deployed, **When** the J15 Accessibility User journey executes, **Then** all 7 acceptance checks above pass with at least 15 distinct assertion points and zero axe-core AA violations.
2. **Given** an axe-core violation is found, **When** the journey runs in CI, **Then** the journey fails the CI gate with a violation report attached — the journey serves as constitution rule 28's executable enforcement.
3. **Given** a per-page allowlist exists for a documented exemption (e.g., a third-party widget with a known issue), **When** the journey runs, **Then** the allowlist requires a written justification and a remediation tracking ID — silent waivers MUST NOT be accepted.

### User Story 9 - Compliance Auditor Exports and Verifies an Audit Trail (Priority: P2)

A Compliance Auditor preparing for a regulatory audit MUST be able to request an audit trail export for a specific period, verify the export's cryptographic signature using the published public key, verify the hash chain integrity end-to-end, query specific event types (authentication, DSR, policy violations, JIT grants), review the compliance evidence dashboard for SOC2/ISO27001 controls, and export a single evidence bundle for regulatory submission. The Compliance Auditor journey (J16) MUST exercise this flow against the real `audit/` (extended by UPD-024) and `security_compliance/` (UPD-024) bounded contexts.

**Why this priority**: Compliance audits are scheduled, not on-demand, so the journey is P2 (run in CI but not on every PR). The functional contract is critical, but the time-to-test is less acute than the four hot-path P1 governance journeys (J10-J13).

**Independent Test**: Request an audit export for a 30-day window via the admin API, verify the export's signature using the public key, verify the chain-integrity end-to-end, query each of the four event types, review the compliance evidence dashboard, and export the evidence bundle. Verify (a) the export is generated with a cryptographic signature; (b) external signature verification succeeds using the published public key; (c) hash chain verification returns ✓ end-to-end; (d) event queries return expected counts and types; (e) the compliance evidence dashboard shows current SOC2/ISO27001 control state; (f) the evidence bundle is downloadable as a single archive.

**Acceptance Scenarios**:

1. **Given** the platform has 30 days of seeded audit history, **When** the J16 Compliance Auditor journey executes, **Then** all 6 stages pass with at least 16 distinct assertion points.
2. **Given** the chain integrity check finds a broken link (test fault-injection variant), **When** the auditor runs the verification, **Then** the verification returns ✗ with the broken-at sequence number and the journey asserts this negative path.
3. **Given** an evidence-bundle export is requested, **When** the bundle is generated, **Then** every artifact in the bundle is referenced by its hash in a manifest signed by the same key — the journey asserts the manifest is internally consistent.

### User Story 10 - Maintainer Adds and Maintains BC-Specific E2E Suites for the New Bounded Contexts (Priority: P2)

A maintainer extending platform coverage MUST be able to add E2E test files inside `tests/e2e/suites/<new-bc>/` for each new audit-pass bounded context (`privacy_compliance/`, `security_compliance/`, `cost_governance/`, `multi_region_ops/`, `model_catalog/`, `localization/`) using the same fixtures and conventions as the existing suites. ### Edge Cases

- **Loki unreachable at journey start**: J17 (and any journey that asserts a Loki log presence) MUST fail with a clear diagnostic ("Loki at platform-observability/loki:3100 returned 503; aborting log assertion") and MUST NOT silently pass. The `assert_log_contains` helper MUST verify Loki reachability before its first poll.
- **Observability stack not yet ready when journeys start**: The journey harness's setUp MUST wait up to a configured deadline (default 60s) for Grafana's `/api/health` endpoint, Prometheus's `/-/ready` endpoint, Loki's `/ready` endpoint, and Jaeger's collector port to all return success before any journey executes. If the deadline elapses, the entire journey run aborts with a clear stack-not-ready diagnostic.
- **Cascade deletion partially fails (J10)**: The Privacy Officer journey MUST cover both the success path (full cascade succeeds, tombstone created, DSR `completed`) and the negative path (one store deletion fails, tombstone NOT created, DSR `failed`); both are explicit assertion points, not conditional.
- **Failover mid-test pod kill (J13)**: The SRE journey MUST verify that the secondary region is `ready` before initiating failover; if it is not, the journey fails fast with a capacity-error diagnostic rather than timing out on a stuck failover.
- **axe-core false positives or known-issue waivers (J15)**: The accessibility journey MUST require an explicit, version-controlled allowlist file for any AA violation that is suppressed; each entry MUST carry (a) a written justification, (b) a remediation tracking ID, (c) an expiry date. Silent waivers MUST NOT be accepted, and the CI gate MUST fail on any violation that is NOT in the allowlist.
- **Helm uninstall leaves orphans**: The umbrella chart's `helm uninstall` MUST be tested explicitly (T058) — the test asserts that no CRDs, no PVCs labelled by this chart, no webhooks, and no ConfigMaps from the chart remain in the namespace. If the chart's sub-charts ship CRDs that survive uninstall (typical for Prometheus Operator and Jaeger Operator), the chart's README MUST document the manual-cleanup steps and the test MUST verify the documentation matches reality.
- **Sub-chart version drift**: The umbrella chart's `Chart.lock` MUST be committed to the repository so that the version-pinned sub-chart resolution is reproducible. CI MUST fail if `Chart.yaml` declares a sub-chart range that does not match the lock.
- **Cross-cluster journey path (managed cluster)**: The chart's `standard` and `enterprise` presets MUST be tested at least once on a real managed cluster (GKE/EKS/AKS) per FR-514. The kind cluster CI run is the everyday gate; the managed-cluster run is a periodic (e.g., weekly) gate. Both runs MUST exist.
- **Grafana SSO failure**: If SSO is configured but unreachable, Grafana MUST refuse access (no fallback to local Grafana auth). The umbrella chart's values MUST NOT permit a fallback path; the test MUST assert this.
- **Frontend rebuild between e2e runs**: Journey tests that exercise the frontend (J15 in particular) MUST be parameterised on the deployed frontend image tag so that journey runs against a stale image fail with a clear image-mismatch diagnostic rather than a timing-related flake.
