> 12. **Cache TTL and max staleness are platform-tunable.** The brownfield proposes `cache.ttlSeconds: 60` / `cache.maxStalenessSeconds: 300` defaults. UPD-040 honours these defaults. The cache is per-pod (in-process LRU); flushable via `platform-cli vault flush-cache --pod=...` or via the admin UI's cache-flush action. Cluster-wide cache flushes broadcast via the existing Kafka `platform.events` topic; each pod re-fills its cache lazily on next read.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Production Operator Installs Musematic with HA Vault (Priority: P1)

A platform operator deploys Musematic to production on Hetzner per UPD-039's FR-608 installation guide. They want HashiCorp Vault as the secret backend in HA mode (3 Raft replicas) co-located in the same Kubernetes cluster. The operator chooses Vault during the install (via `helm install` values OR via `platform-cli install --vault-mode=ha`) and the platform's control plane authenticates to Vault via Kubernetes auth using its ServiceAccount. Once running, every credential the platform needs (OAuth client secrets, model provider API keys, connector credentials, notifications signing secrets, IBOR connector credentials) is resolved through Vault. The operator never sees plaintext credentials in any log, config map, or Kubernetes Secret.

**Why this priority**: Rule 10 ("Every credential goes through vault") is constitutionally binding, and today it is unenforceable because no real Vault implementation exists. P1 because (a) the audit-pass roster names UPD-040 alongside UPD-035-UPD-039 as the v1.3.0 cohort; (b) without it, every other bounded context's secret-resolution path is unsafe by construction (env-var fallback paths in OAuth, notifications, model_catalog, IBOR); (c) production self-hosted customers cannot ship without a real Vault per the security-guide FR-618 contract; (d) the FR-688 fail-safe behaviour (no hardcoded fallback) cannot be honoured until the `vault` mode is actually implemented.

**Independent Test**: A fresh Kubernetes cluster (kind-of-3-nodes OR a real Hetzner cluster following the FR-608 guide) is provisioned with `vault.mode=ha` and `vault.authMethod=kubernetes`. Verify (a) 3 Vault pods reach `Running` status with Raft cluster initialized within 5 minutes; (b) Vault is unsealed (auto-unseal via Hetzner KMS / cloud KMS / manual Shamir shares ŌĆö operator-chosen); (c) the platform's control plane pod authenticates to Vault on startup via its ServiceAccount and emits a structured-log entry `{event: "vault.authenticated", auth_method: "kubernetes", token_expiry_at: "..."}` (no token value logged); (d) a synthetic write at `secret/data/musematic/{env}/test/value` is readable by the control plane within 1 second; (e) lease renewal fires at 50% of token TTL with a corresponding `{event: "vault.lease_renewed"}` log entry; (f) Prometheus scrapes `vault_lease_count`, `vault_renewal_success_total`, `vault_auth_failure_total`, `vault_read_total`, `vault_cache_hit_ratio` metrics from each control-plane pod; (g) a synthetic OAuth provider configured with `client_secret_ref=secret/data/musematic/{env}/oauth/google` has its secret resolved on the next OAuth callback (verified by a test login flow); (h) `/admin/security/vault` (delivered by UPD-036) shows `status=green`, `auth_method=kubernetes`, `cache_hit_rate>0%`, `recent_failures=0`; (i) `kubectl logs platform-control-plane-...` contains zero plaintext-secret regex matches (`API_KEY=...`, `password=...`, etc.) per Rule 10's no-plaintext-in-logs invariant.

**Acceptance Scenarios**:

1. **Given** a fresh Kubernetes cluster, **When** the operator runs `helm install platform deploy/helm/platform/ --set vault.mode=ha --set vault.authMethod=kubernetes`, **Then** 3 Vault pods reach `Running` status, the platform pod authenticates via its ServiceAccount, and a synthetic test secret is readable within 1 second of write.
2. **Given** the platform is running with `vault.mode=ha`, **When** the operator restarts a control-plane pod, **Then** the new pod re-authenticates to Vault on startup without operator intervention; no manual token re-issuance is required.
3. **Given** the platform's token TTL is 1 hour, **When** 30 minutes elapse (50% of TTL), **Then** the platform emits a lease-renewal request and the new token TTL re-extends to 1 hour; the `vault_renewal_success_total` counter increments.
4. **Given** an OAuth provider's `client_secret_ref` points at `secret/data/musematic/production/oauth/google`, **When** a user completes the Google OAuth callback, **Then** the platform resolves the secret via the new `SecretProvider` flow (NOT via the bespoke env-var fallback path that exists today at `oauth_service.py:732-747`), and the OAuth flow succeeds.
5. **Given** the operator inspects `kubectl logs platform-control-plane-...` after 24 hours of operation, **When** they grep for the canonical secret-leak regex set (passwords, API keys, JWT tokens, Bearer tokens), **Then** zero matches are returned.

### User Story 2 - Operator Migrates from Kubernetes Secrets to Vault Without Downtime (Priority: P1)

An operator runs Musematic today on a production cluster using Kubernetes Secrets (the transitional `kubernetes` mode introduced by this feature) ŌĆö they have not yet adopted Vault. They want to migrate to Vault without any service interruption. They install the Vault sub-chart alongside the running platform, run `platform-cli vault migrate-from-k8s`, verify the migration manifest, flip `PLATFORM_VAULT_MODE=vault` via a Helm upgrade, and observe zero auth/login failures during the cutover.

**Why this priority**: Existing self-hosted deployments (those that landed before UPD-040) MUST have a non-disruptive upgrade path. P1 because (a) without the migration tool, every existing deployment requires a full re-install OR a manual `vault kv put` for every secret (operator-error-prone, easily 50+ secrets to migrate); (b) constitutional Rule 10 cannot be retroactively enforced on existing deployments without this tool; (c) the audit-pass roster's value proposition for v1.3.0 hinges on every constitutional rule being achievable on every deployment, including upgrades.

**Independent Test**: A pre-existing Musematic deployment running with `PLATFORM_VAULT_MODE=kubernetes` (and Kubernetes Secrets populated for auth, model_catalog, notifications, IBOR per the canonical path scheme `secret/data/musematic/{env}/{domain}/{resource}` mirrored in K8s as `Secret name=musematic-{env}-{domain}-{resource}`). Operator runs (a) `helm install vault deploy/helm/vault/ --set mode=ha`; (b) `platform-cli vault migrate-from-k8s --env=production --dry-run`; (c) verifies the dry-run manifest output lists every K8s Secret ŌåÆ Vault path mapping with `dry_run: true`; (d) `platform-cli vault migrate-from-k8s --env=production` (no dry-run); (e) verifies the migration manifest at `vault-migration-{timestamp}.json` includes `success_count=N`, `failure_count=0`, and a SHA-256 hash of each value (NOT the value itself); (f) `helm upgrade platform deploy/helm/platform/ --set vault.mode=ha` (the cutover); (g) monitors `vault_auth_failure_total` over the next 60 minutes ŌĆö must stay zero; (h) re-runs `platform-cli vault migrate-from-k8s --env=production` (idempotency test) ŌĆö must report `already_migrated_count=N, new_count=0`; (i) flips back to `PLATFORM_VAULT_MODE=kubernetes` (rollback verification) ŌĆö verifies the platform resumes reading from K8s with zero auth failures.

**Acceptance Scenarios**:

1. **Given** an operator on a pre-Vault deployment with `N` Kubernetes Secrets containing platform credentials, **When** they run `platform-cli vault migrate-from-k8s`, **Then** every Secret is mirrored at the canonical Vault path and the migration manifest documents source ŌåÆ destination ŌåÆ success.
2. **Given** the migration completes and the operator flips `PLATFORM_VAULT_MODE=vault`, **When** users log in / agents resolve credentials over the next 60 minutes, **Then** zero authentication failures attributable to the cutover are observed.
3. **Given** the migration was run yesterday and a new K8s Secret is added today, **When** the operator re-runs `platform-cli vault migrate-from-k8s`, **Then** the tool migrates only the new Secret and reports `already_migrated_count=N, new_count=1`.
4. **Given** the operator wants to roll back to Kubernetes Secrets (post-migration but before deleting them), **When** they flip `PLATFORM_VAULT_MODE=kubernetes` and `helm upgrade`, **Then** the platform resumes reading from K8s with zero failures (the Secrets remain in the cluster ŌĆö UPD-040 never deletes them).
5. **Given** the migration manifest is committed to the operator's audit log, **When** an auditor verifies post-hoc, **Then** the manifest includes per-secret SHA-256 hashes (NOT plaintext values) and the manifest passes a structural-validity check via `platform-cli vault verify-migration --manifest=vault-migration-2026-04-27.json`.

### User Story 3 - Security Engineer Rotates a Model Provider API Key via Vault (Priority: P2)

A security engineer schedules quarterly rotation of the OpenAI API key per UPD-024's secret_rotation_service. The rotation creates a new Vault version of `secret/data/musematic/production/model-providers/openai`, the platform's existing executions in flight continue using the prior version (cache TTL bridges the dual-credential window), new executions use the new version after cache TTL elapses, and the prior version is destroyed via `vault kv metadata delete` after the configured dual-credential window. The audit chain records the rotation with non-secret metadata only (the rotation request, the timestamp, the new-version metadata) ŌĆö no key value appears in any log.

**Why this priority**: Secret rotation per UPD-024 is constitutionally important (Rule 10 references the rotation service explicitly) but is P2 because (a) the rotation flow already works today via UPD-024's `RotatableSecretProvider` (which UPD-040 rewires onto the consolidated Protocol per spec correction ┬¦3); (b) the change UPD-040 introduces is removing UPD-024's bespoke dual-credential window logic in favour of Vault's native KV v2 versioning ŌĆö the rotation IS still functional pre-UPD-040 via the bespoke logic; (c) the User Stories 1 and 2 must land first to deliver any value.

**Independent Test**: A test deployment with a populated `secret/data/musematic/test/model-providers/openai` (version 1). Operator triggers `platform-cli admin rotate-secret --target=model-providers/openai --new-value-from-stdin`; verifies (a) Vault version 2 is created via `vault kv put`; (b) the prior version 1 is RETAINED for the configured dual-credential window (default 60 seconds ŌĆö matching cache TTL); (c) executions started in the first 60 seconds continue using version 1 (read from the per-pod cache); (d) executions started after 60 seconds resolve version 2 (cache miss ŌåÆ fresh Vault read); (e) after the dual-credential window elapses, `vault kv metadata delete` is invoked and version 1 is permanently removed; (f) the audit chain entry for the rotation contains `{action: "secret_rotated", target: "model-providers/openai", old_version: 1, new_version: 2}` with NO key value; (g) Loki/Promtail dashboards show zero plaintext-secret hits during the rotation window.

**Acceptance Scenarios**:

1. **Given** a populated KV v2 entry at `secret/data/musematic/{env}/model-providers/openai` (v1), **When** the operator triggers a rotation, **Then** v2 is written, both versions are readable for the dual-credential window, and v1 is destroyed at window expiry.
2. **Given** a long-running execution started at t=0 with version 1 in its per-pod cache, **When** the rotation occurs at t=10s and the cache TTL is 60s, **Then** the execution continues using v1 until t=60s and seamlessly transitions to v2 thereafter.
3. **Given** the rotation completes, **When** the security engineer reviews the audit chain, **Then** the rotation entry documents the version transition without exposing the key value.

### User Story 4 - Super Admin Verifies Vault Status During an Incident (Priority: P2)

During an incident (e.g., suspected credential leak, OAuth outage, rate-limit storms from a model provider), a super admin needs to quickly verify Vault is healthy and to flush the per-pod cache to force fresh reads. They open `/admin/security/vault` (delivered by UPD-036; consumes UPD-040's backend contracts), inspect the connection / auth / lease panels, and trigger a manual cache flush. Subsequent reads hit Vault directly (cache miss); the admin can confirm the platform is now reading the latest values.

**Why this priority**: Incident response is constitutionally important (Rule 30 ŌĆö admin role gates; Rule 31 ŌĆö bootstrap secrets never logged; UPD-036's incident-runbook scope) but P2 because the data is reachable via `platform-cli vault status` as a CLI fallback if the UI page is delayed. The CLI path is the MVP; the UI page is the full-feature path delivered by UPD-036.

**Independent Test**: An operator opens the page (or runs `platform-cli vault status`) and verifies (a) connection status (`green` / `yellow` / `red`); (b) auth method (`kubernetes` / `approle` / `token`); (c) token expiry timestamp; (d) lease count; (e) recent auth failures (last 10, with timestamps); (f) per-BC read rates (auth, model_catalog, notifications, IBOR); (g) cache hit rate (rolling 5-minute window). Operator triggers `platform-cli vault flush-cache --pod=platform-control-plane-0` (or the UI's cache-flush button in UPD-036); verifies the next read on that pod is a cache miss (visible via the `vault_read_total` metric incrementing without a `vault_cache_hit_total` increment).

**Acceptance Scenarios**:

1. **Given** the platform is running with Vault healthy, **When** an admin loads `/admin/security/vault` (or runs `platform-cli vault status`), **Then** all panels render within 3 seconds and reflect the current state.
2. **Given** the admin triggers a cache flush, **When** the next request requiring a credential arrives, **Then** the credential is resolved via a fresh Vault read (cache miss verified by metrics).
3. **Given** a recent auth failure occurred, **When** the admin clicks the failure entry, **Then** the entry links to the corresponding Loki log line (via the Loki labels established by UPD-084 ŌĆö log aggregation and dashboards).

### User Story 5 - Platform Degrades Gracefully When Vault Is Unreachable (Priority: P1)

During a network partition between the platform and Vault (e.g., misconfigured NetworkPolicy, Vault pod scheduler eviction, a bug in the Hetzner Cloud LB configuration during failover), the platform must NOT bypass authentication. It must (a) continue serving cached credentials within the cache TTL window; (b) optionally serve stale credentials past the TTL up to a configured `maxStalenessSeconds` ceiling; (c) refuse critical reads (login, OAuth callback, IBOR sync) explicitly when the cache is cold AND Vault is unreachable; (d) NEVER use a hardcoded fallback path; (e) emit an alert ("Vault unreachable") within 1 minute; (f) recover lazily on the next successful read after Vault is restored.

**Why this priority**: This is constitutionally binding (Rule 10 ŌĆö no plaintext credentials in code/config; the failure mode MUST refuse, not bypass). P1 because (a) the failure-mode design is part of the security contract for Rule 10; (b) without it, a Vault outage degrades into a credential-bypass risk; (c) the audit-pass roster requires every `mode` (mock / kubernetes / vault) to be safe under failure.

**Independent Test**: A test deployment with Vault in `vault.mode=ha`. Operator (a) populates the cache by running 100 synthetic reads (verified by `vault_cache_hit_total`); (b) blocks network egress from platform pods to Vault (e.g., `kubectl apply` a NetworkPolicy denying egress to the Vault service); (c) verifies cached reads continue serving for the next 60 seconds (cache TTL); (d) at t=60s, verifies stale reads serve up to `maxStalenessSeconds=300s` with a structured-log entry `{event: "vault.serving_stale", path: "...", stale_age_seconds: 65}`; (e) at t=300s, verifies critical reads (e.g., a fresh OAuth callback for a previously-uncached provider) FAIL with a clear error (HTTP 503 + structured log; the user sees "Authentication system unavailable. Please try again."); (f) verifies the `vault_unreachable_alert` Prometheus alert fires within 60 seconds of the partition; (g) restores network; verifies the next successful read repopulates the cache lazily and the alert clears within 5 minutes.

**Acceptance Scenarios**:

1. **Given** Vault is reachable and the cache is warm, **When** the platform receives a credential-resolve request, **Then** the cached value is served within 5ms (cache hit metric increments).
2. **Given** Vault becomes unreachable but the cache is warm, **When** the platform receives a credential-resolve request within the cache TTL, **Then** the cached value is served and the underlying Vault failure is logged but NOT surfaced to the user.
3. **Given** Vault has been unreachable for `maxStalenessSeconds` AND the cache is cold for a specific path, **When** the platform receives a critical-path request (login, OAuth callback), **Then** the request fails with a clear error and the user sees a degraded-state message; NO hardcoded fallback path is invoked.
4. **Given** the partition has been observed for 1 minute, **When** Prometheus evaluates alert rules, **Then** the `VaultUnreachable` alert fires.
5. **Given** Vault is restored, **When** the next read attempt succeeds, **Then** the cache is repopulated lazily and the alert clears within 5 minutes.

### Edge Cases

- **Kubernetes auth ServiceAccount token rotation mid-request.** Kubernetes rotates ServiceAccount tokens every few hours; the Vault Kubernetes auth method MUST handle this transparently ŌĆö the next failed Vault request triggers re-authentication using the fresh on-disk token at `serviceAccountTokenPath`.
- **AppRole SecretID expired during operation.** The platform detects a Vault 403 for an AppRole-authenticated request, attempts re-authentication using the on-disk SecretID file (refreshed by an external rotator ŌĆö operator-managed); fails with a clear error if the SecretID file is itself stale.
- **Vault policy denies a path that the platform attempts to read.** The Vault client receives a 403; emits a structured-log entry `{event: "vault.policy_denied", path: "..."}`, fails the operation, and NEVER retries with a different path. This catches misconfigured policies early without silently masking permission errors.
- **KV v2 version retention exceeded.** Vault's own retention policy auto-destroys versions beyond `max_versions`; UPD-040 honours this and treats the resulting "version not found" error as a permanent failure (NOT a retryable condition).
- **Vault audit log disk pressure.** If Vault's audit log device is full, Vault refuses writes per its own design. UPD-040 surfaces this as `{event: "vault.audit_full"}` and emits an alert; reads continue to function, but writes (rotation, migration) fail until the operator clears audit log space.
- **Clock skew between platform and Vault.** Token expiry timestamps are computed by Vault and trusted by the platform. If clock skew exceeds 30 seconds (detected via the `Date` HTTP response header), the platform refreshes tokens more conservatively (renewal at 40% TTL instead of 50%) and emits a structured-log warning.
- **Operator sets `PLATFORM_VAULT_AUTH_METHOD=token` in production.** Per Rule 10 and the brownfield's edge case, the platform refuses to start unless `ALLOW_INSECURE=true` AND `PLATFORM_ENVIRONMENT=dev` are both set. The startup error message points at the constitutional rule and the migration runbook (FR-617).
- **Migration tool encounters a Kubernetes Secret without a clear canonical Vault path mapping.** The migration tool emits a `{event: "vault.migration.unmapped", k8s_secret: "..."}` entry to the manifest with `success: false, reason: "no_canonical_path"`, and continues. The operator post-edits the manifest and re-runs.
- **Two pods race on the same KV v2 write (e.g., dual rotators).** Vault's CAS (check-and-set) parameter on `vault kv put` prevents lost writes; UPD-040 uses CAS on every write triggered by `RotatableSecretProvider` and on the migration tool's writes. Conflicts result in a clear `{event: "vault.write_cas_conflict"}` log + a retry with the latest version (max 3 attempts before failure).
- **Cache flush is requested while Vault is unreachable.** The cache-flush admin action succeeds locally (clears the in-process LRU); subsequent reads will fail until Vault is restored. The action emits `{event: "vault.cache_flushed_during_outage", warning: true}` so operators understand the implication.
- **Existing `mock` mode behaviour regression.** The mock-mode resolver path at `security.py:60-82` MUST continue working unchanged (read from `.vault-secrets.json` + env-var fallback) for kind / dev / CI flows. UPD-040's regression test pinpoints any change in `_resolve_mock` semantics.

## Assumptions

- **HashiCorp Vault is the chosen secret backend; alternatives (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault) are out of scope.** The audit-pass roster names "HashiCorp Vault Integration" explicitly. Cloud-provider secret managers MAY be added in a future feature via additional `SecretProvider` implementations; this feature does NOT design for that.
- **The upstream `hashicorp/vault` Helm chart is permissively licensed (MPL 2.0) and acceptable for a wrapper sub-chart.** The plan phase pins the exact version.
- **`hvac>=2.3.0` (Python) and `github.com/hashicorp/vault/api` (Go, latest stable) are acceptable runtime dependencies.** Both are MPL 2.0.
- **The platform's deployment topology (modular monolith Python control plane + Go satellites + Next.js frontend) does NOT change.** Vault is a NEW infrastructure dependency, not an architectural shift.
- **Auto-unseal is operator-chosen.** UPD-040 documents three options in the operator guide: (a) cloud KMS auto-unseal (Hetzner / GCP / AWS / Azure), (b) Transit auto-unseal via a separate Vault cluster (advanced), (c) manual Shamir-share unseal (default for `dev` / `standalone` presets); the choice is operator-driven and out of scope for the platform code itself.
- **Existing Kubernetes Secrets (in `kubernetes` mode) follow the canonical-path-mirrored Secret-name scheme** (`Secret name=musematic-{env}-{domain}-{resource}`, key=value). If an operator's pre-existing deployment uses a different scheme, they MUST rename their Secrets before migration OR provide a custom mapping config to `platform-cli vault migrate-from-k8s --mapping=path/to/mapping.json`.
- **The `RotatableSecretProvider` (UPD-024) public method names are STABLE and MUST be preserved** when rewired onto the new Protocol; UPD-024's tests are the canonical contract.
- **The `notifications/channel_router.py:45-48` `SecretProvider` import compatibility shim is removed in v1.4.0** (one release after UPD-040). All callsites must move to `common.secret_provider` by then.
- **UPD-036's `/admin/security/vault` UI page** consumes UPD-040's three backend endpoints; if UPD-036 has not landed yet, UPD-040 ships `platform-cli vault status` as the CLI fallback per User Story 4.
- **UPD-039's `scripts/generate-env-docs.py`** is the integration point for the FR-687 secret-pattern deny-list check; if UPD-039 has not landed at the time of UPD-040 implementation, UPD-040 commits a standalone `scripts/check-secret-patterns.py` that is later merged into UPD-039's script.
- **UPD-024's secret_rotation_service** is the integration point for FR-692; UPD-040 does NOT modify the rotation policy, only the underlying storage path.
- **UPD-047's observability bundle** is the home for the Prometheus alert rules (FR-697 metrics, FR-688 alerts); UPD-040 commits the alert YAMLs under `deploy/helm/observability/templates/alerts/vault.yaml`.

- **Out of scope:**
  - **Vault Enterprise features** (namespaces beyond a single cluster, replication, Sentinel policies, control groups). The brownfield's `namespace: ""` field is reserved for Enterprise users; the open-source build is the supported path.
  - **Vault Agent (sidecar / init-container injector pattern).** UPD-040 explicitly disables the injector (`vault.injector.enabled: false`); the platform uses `hvac` / `vault-api` clients directly.
  - **Cloud-provider secret manager backends.** Future work; not in this feature.

  - **Auto-rotation of the platform's own ServiceAccount token.** Kubernetes manages SA token rotation; UPD-040 only ensures the Vault client handles the rotation transparently.
  - **Migration FROM Vault TO Kubernetes Secrets.** The migration tool is one-way; reverse migration is achievable by manually exporting Vault values into K8s Secrets but is not a supported tool.
