## User Scenarios & Testing *(mandatory)*

### User Story 1 - Workspace Owner Reviews Their Workspace (Priority: P1)

A workspace owner logs in to review the current state of their team's work. The dashboard at `/workspaces/{id}` (NEW route — does NOT exist today per spec correction §9) renders within 3 seconds with: active goals count, executions in flight, agent count, budget gauge (driven by the existing `cost_budget` JSONB on `WorkspaceSettings` per spec correction §12 + UPD-079 cost_governance metrics), quota usage per resource (agents/fleets/executions/storage from the NEW `quota_config` JSONB per spec correction §14), tag summary (UPD-033 reuse per spec correction §15), DLP violations count (UPD-076/078 reuse), recent audit activity scoped to the workspace. All data scoped to this workspace only — no leakage from other workspaces.

**Why this priority**: Per FR-658 + Rule 45, P1 because (a) the workspace owner has NO dedicated dashboard today (verified per spec correction §9 — no `/workspaces/[id]/*` UI route exists); (b) the existing admin pages at `/admin/settings?tab=workspaces` (delivered by UPD-036 — research §8 confirms tab pattern) are admin-global views, NOT workspace-owner-scoped; (c) every other user story in UPD-043 navigates from this dashboard — without it, the workbench is incoherent.

**Independent Test**: Test user is the owner of workspace `WS_001` containing 3 active goals + 5 in-flight executions + 12 agents + tags + a `cost_budget` of `$500/month` at 60% consumption. User logs in; navigates to `/workspaces/WS_001`. Verifies (a) dashboard loads in ≤ 3 seconds; (b) all 5 cards render real data (counts match the seeded fixtures); (c) budget gauge shows 60% consumption with a forecast for end-of-period spend (driven by UPD-079's existing forecast service per research §12); (d) quota usage bars show actual usage (e.g., "12 of 50 agents", "5 of 100 executions"); (e) DLP violations count matches the audit-chain entries with `event_type='privacy.dlp.violation' AND workspace_id=WS_001`; (f) recent activity feed shows the last 10 audit entries for the workspace; (g) navigates to a sibling workspace `WS_002` (where the user is NOT a member) — backend rejects with 403 per the existing service-layer permission check.

**Acceptance Scenarios**:

1. **Given** a workspace owner with seeded data, **When** they navigate to `/workspaces/{id}`, **Then** the dashboard loads in ≤ 3 seconds with all cards populated.
2. **Given** the owner has visibility grants from external resources, **When** the dashboard renders, **Then** visibility-dependent metrics (cross-workspace agents accessible) are populated; without grants, those metrics show zero with a clear "no external visibility configured" tooltip per spec edge case.
3. **Given** the user is NOT a member of the requested workspace, **When** they navigate to `/workspaces/{id}`, **Then** the existing service-layer permission check returns 403; the UI shows a clear "access denied" page.
4. **Given** the workspace's `cost_budget` is at 80%, **When** the dashboard loads, **Then** the budget gauge renders with a yellow warning indicator (driven by UPD-079's threshold logic).
5. **Given** the workspace has `quota_config={agents: 10, fleets: 5, executions: 100, storage_gb: 50}` AND actual usage is `{agents: 9, fleets: 5, executions: 98, storage_gb: 45}`, **When** the dashboard loads, **Then** quota bars render with appropriate color coding (green / yellow / red).

### User Story 2 - Workspace Owner Invites a New Member (Priority: P1)

A workspace owner invites a new team member by email. The invitation flows through UPD-077 multi-channel notifications (email channel); on acceptance, the user appears in the list. Ownership transfer requires 2PA per Rule 33 + FR-561 (verified canonical contract per research §15).

**Why this priority**: Per FR-659 + Rule 45, P1 because (a) member management is the canonical workspace-owner task; (b) the existing backend at `workspaces/router.py:91-121` (POST/PATCH/DELETE on `/members`) works today but has NO UI surface; (c) the role-picker UI is the visible expression of the existing `WorkspaceRole` enum.

**Independent Test**: Test user is the owner of `WS_001` with 3 existing members. User navigates to `/workspaces/WS_001/members`. Verifies (a) the page lists 3 existing members with role + joined-date + recent-activity link; (b) clicks "Invite member"; modal opens with email input + role select (4 options) + role descriptions; (c) submits with `email=alice@example.com, role=member`; backend calls `POST /workspaces/WS_001/members` (existing endpoint); the invitation email is sent via UPD-077 email deliverer; (d) `alice@example.com` opens the email + clicks accept; she appears in the member list; (e) the audit chain has `auth.workspace.member_added` entry; (f) owner clicks "Transfer ownership" on a co-owner — 2PA challenge dialog opens per spec correction §12 + Rule 33 (the 2PA challenge requires an admin co-signer to approve before the transfer commits); after co-signer approval, ownership transfers; the audit chain has BOTH `auth.workspace.transfer_initiated` (single-actor) and `auth.workspace.transfer_committed` (post-2PA) entries; (g) clicks "Remove" on a member; backend calls existing `DELETE /workspaces/WS_001/members/{user_id}` endpoint; member's access revoked; audit entry emitted.

**Acceptance Scenarios**:

1. **Given** a workspace with 3 existing members, **When** the owner navigates to `/members`, **Then** all 3 members render with role + joined-date.
2. **Given** the owner invites a new member by email, **When** the form submits, **Then** the existing `POST /members` endpoint is called + an invitation email is sent via UPD-077.
3. **Given** the owner clicks "Transfer ownership", **When** the action requires 2PA per Rule 33 + FR-561, **Then** a 2PA challenge dialog opens; an admin co-signer approves; the transfer commits with double-audit (one entry per actor).
4. **Given** the owner clicks "Remove member", **When** the action confirms, **Then** the existing `DELETE /members/{user_id}` endpoint is called + access is revoked + audit entry emitted.
5. **Given** an in-flight execution belongs to the removed member, **When** the removal commits, **Then** the in-flight execution continues to completion (per spec edge case); the member loses read access on subsequent navigation.

### Edge Cases

- **Owner without visibility grants**: dashboard still loads; visibility-dependent metrics show zero with a clear "no external visibility configured" tooltip; the visibility explorer renders the workspace as an isolated node with "deny all" badge.
- **Member removed mid-execution**: in-flight executions continue to completion (the existing executions BC does not check membership at runtime); the member loses read access on subsequent navigation per the existing service-layer permission check.
- **Ownership transfer while member invitation pending**: the invitation row stays valid; the new owner sees pending invitations on `/members`; the existing `Membership` model has no FK to the inviter.
- **Workspace archive with active executions**: existing `POST /workspaces/{id}/archive` endpoint is gated by a typed-confirmation per FR-660; the new UI surface adds a clear warning banner listing active executions; option to cancel them first.
- **Connector credentials invalid**: the connector's `enabled` flag is auto-flipped to `false` by the existing health-check infrastructure; an alert fires to the workspace owner via UPD-077; agents fall back to alternative connectors (existing routing logic) or emit attention requests.
- **Connector rate-limited by third party (Slack 429)**: the existing `outbound_deliveries` table (per UPD-077) records the 429 + retry counts; the activity panel surfaces this; alerts the workspace owner if retry attempts exceed the configured threshold.
- **LDAP connector times out**: the existing `IBORSyncRun` row records the timeout error; the new `GET /admin/ibor/{id}/sync-history` endpoint returns the row; the UI links to Loki logs per UPD-084 integration.
- **2PA challenge expires**: the new `TwoPersonApprovalChallenge` table per spec correction §12 has a TTL (default 5 minutes); on expiry, the challenge is invalidated; the user must re-initiate.
- **2PA admin co-signer is the same user as the initiator**: the backend rejects with HTTP 400 + clear error "the co-signer must be a different user" per Rule 33's "two-person" requirement.
- **Test-connectivity sends a user-visible message inadvertently**: the implementation MUST use connector-specific dry-run paths (Slack `auth.test`, Telegram `getMe`, Webhook HEAD request) — NOT user-channel writes. Verified by SC-006 + dedicated E2E test.
- **Workspace owner deletes themselves from members**: backend rejects (the existing `DELETE /members/{user_id}` endpoint validates that the actor is not the workspace owner unless ownership is being transferred first per spec User Story 2 acceptance scenario 5).
- **Quota exceeded**: when usage reaches 100% of `quota_config[resource]`, the platform blocks new resource creation at the resource-creation endpoint (existing pattern from UPD-079 budget; extended to quotas in this feature) with HTTP 402 + clear error.
