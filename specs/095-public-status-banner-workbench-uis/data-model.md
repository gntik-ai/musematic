# Data Model — UPD-045 (feature 095)

This document captures the entity model added by UPD-045. All schema changes are delivered via Alembic migration `095_status_page_and_scenarios.py` (subject to slot re-confirmation at tasks time per research R5). Existing entities are referenced but not rewritten.

---

## 1. PlatformStatusSnapshot (new)

**Table**: `platform_status_snapshots`

The composed roll-up that backs both the public status surface and the in-shell banner.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | UUID | PK, default `gen_random_uuid()` | |
| `generated_at` | timestamptz | NOT NULL, default `now()`, INDEX | The snapshot's logical time. Used for 30-day rollup and "last known good" age display. |
| `overall_state` | varchar(32) | NOT NULL, CHECK in `('operational','degraded','partial_outage','full_outage','maintenance')` | First-viewport summary value. |
| `payload` | JSONB | NOT NULL | See "Snapshot payload schema" below. |
| `source_kind` | varchar(32) | NOT NULL, CHECK in `('kafka','poll','fallback','manual')` | Provenance for debugging. |
| `created_by` | UUID | NULL, FK `users(id)` ON DELETE SET NULL | NULL for system-generated snapshots; populated for manual operator overrides. |

**Indexes**:
- `IX_platform_status_snapshots_generated_at_desc` — covering index for "recent snapshots" queries.
- Partial index on `overall_state != 'operational'` for incident-history queries.

**Retention**: 90 days hot in PostgreSQL. Cold archive to S3-compatible storage (via the existing nightly archival pattern) beyond that. The 30-day uptime rollup table is generated by the APScheduler daily job (see `projections.py`).

### Snapshot payload schema (JSONB)

```jsonc
{
  "overall_state": "degraded",
  "components": [
    {
      "id": "control-plane-api",
      "name": "Control Plane API",
      "state": "operational",
      "last_check_at": "2026-04-28T13:45:02Z",
      "uptime_30d_pct": 99.97
    }
    // ... one per component
  ],
  "active_incidents": [
    {
      "id": "incident-uuid",
      "title": "Elevated error rate on control-plane reads",
      "severity": "warning",         // critical|warning|info — mirrors incident_response.models.Incident.severity
      "started_at": "2026-04-28T13:30:00Z",
      "components_affected": ["control-plane-api"],
      "last_update_at": "2026-04-28T13:42:11Z",
      "last_update_summary": "Investigating root cause."
    }
  ],
  "scheduled_maintenance": [
    {
      "window_id": "window-uuid",
      "title": "Q2 platform upgrade",
      "starts_at": "2026-05-02T22:00:00Z",
      "ends_at":   "2026-05-02T23:30:00Z",
      "blocks_writes": true,
      "components_affected": ["control-plane-api","reasoning-engine"]
    }
  ],
  "active_maintenance": null,                            // populated when a window is currently active
  "recently_resolved_incidents": [ /* last 7 days */ ],
  "uptime_30d": {                                        // per-component 30-day uptime
    "control-plane-api": { "pct": 99.97, "incidents": 1 },
    "reasoning-engine":  { "pct": 99.99, "incidents": 0 }
  }
}
```

**Schema enforcement**: The Pydantic schema `PlatformStatusSnapshotPayload` in `status_page/schemas.py` validates the JSON at write time. Reads tolerate forward-compat additions.

## 2. StatusSubscription (new)

**Table**: `status_subscriptions`

Owned by either an anonymous visitor (identified by a confirmation token) or an authenticated user.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | UUID | PK | |
| `channel` | varchar(16) | NOT NULL, CHECK in `('email','rss','atom','webhook','slack')` | RSS/Atom subscriptions are conceptual — the visitor pastes the feed URL into a reader; we record the visitor's identity only if they enter an email for "I want to be notified about changes to the feed schema" (out of scope here). In practice only `email`, `webhook`, `slack` rows are created. RSS/Atom rows reserved for future per-feed subscriber tracking. |
| `target` | text | NOT NULL | Email address / Slack incoming webhook URL. For `webhook` channel see "Webhook subscription wiring" note below. |
| `scope_components` | text[] | NOT NULL, default `'{}'` | Empty array = all components. Otherwise the component IDs the subscription applies to. |
| `confirmation_token_hash` | bytea | NULL | SHA-256 of the confirmation token. Token itself is sent in the confirmation email/Slack message and never stored. NULL for confirmation-not-required channels. |
| `confirmed_at` | timestamptz | NULL | NULL until visitor clicks confirmation link. |
| `health` | varchar(16) | NOT NULL, default `'pending'`, CHECK in `('pending','healthy','unhealthy','unsubscribed')` | `pending` until confirmed; `unhealthy` when an email hard-bounces or a webhook persistently fails. |
| `workspace_id` | UUID | NULL, FK `workspaces(id)` ON DELETE CASCADE | Populated for authenticated subscriptions managed via `/settings/status-subscriptions`. NULL for anonymous-visitor subscriptions. |
| `user_id` | UUID | NULL, FK `users(id)` ON DELETE SET NULL | Populated for authenticated subscriptions. NULL for anonymous. |
| `webhook_id` | UUID | NULL, FK `outbound_webhooks(id)` ON DELETE CASCADE | For `webhook` channel — the subscription delegates to the existing `OutboundWebhook` row (see R7). |
| `created_at` | timestamptz | NOT NULL, default `now()` | |
| `updated_at` | timestamptz | NOT NULL, default `now()`, ON UPDATE | |

**Indexes**:
- UNIQUE on `(channel, target)` partial WHERE `confirmed_at IS NOT NULL` — prevents duplicate confirmed subscriptions per address.
- `IX_status_subscriptions_user_id` for `/settings/status-subscriptions` listing.
- `IX_status_subscriptions_workspace_id` for workspace-scoped admin views.

**State machine**:

```
pending --(token confirmed)--> healthy --(hard-bounce|persistent-fail)--> unhealthy
                                  \--(user/visitor unsubscribes)--> unsubscribed
                                  
unhealthy --(operator marks reviewed/fixed)--> healthy
unsubscribed --(terminal)--> (no further dispatch)
```

**Webhook subscription wiring**: When `channel='webhook'`, this row stores subscription metadata; the actual delivery configuration (URL, signing secret ref, retry policy) lives in `outbound_webhooks` referenced by `webhook_id`. The `OutboundWebhook.event_types` JSONB array is set to the status-page-specific event types (`incident.created`, `incident.updated`, `incident.resolved`, `maintenance.scheduled`, `maintenance.started`, `maintenance.ended`). Anonymous-visitor webhooks attach to the synthetic `system_status` workspace per R7.

## 3. SubscriptionDispatch (new)

**Table**: `subscription_dispatches`

A single dispatch attempt for an event × subscription pair. Append-only.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | UUID | PK | |
| `subscription_id` | UUID | NOT NULL, FK `status_subscriptions(id)` ON DELETE CASCADE | |
| `event_kind` | varchar(48) | NOT NULL, CHECK in (see "Event kinds" below) | |
| `event_id` | UUID | NOT NULL | Source-event identifier (e.g., the `incidents.id` for incident events). Allows joining for cross-checks. |
| `dispatched_at` | timestamptz | NOT NULL, default `now()` | |
| `outcome` | varchar(16) | NOT NULL, CHECK in `('sent','retrying','dead_lettered','dropped')` | `dropped` indicates intentional skip (e.g., scope filter). |
| `webhook_signature_kid` | varchar(64) | NULL | For `webhook` channel: the signing secret KID used (for HMAC verification audit). |
| `error_summary` | text | NULL | First 1KB of the failure detail for ops triage. |

**Indexes**:
- `(subscription_id, dispatched_at DESC)` for per-subscription dispatch history.
- `(event_kind, dispatched_at DESC)` for "did event X get fanned out?" cross-checks.

**Append-only**: Row is INSERT-only. Updates to `outcome` happen via dedicated retry-state rows (delegated to `webhook_deliveries` for `webhook`-channel rows; written here as additional rows for `email`/`slack`).

### Event kinds

```
incident.created
incident.updated
incident.resolved
maintenance.scheduled
maintenance.started
maintenance.ended
component.degraded
component.recovered
```

## 4. SimulationScenario (new)

**Table**: `simulation_scenarios`

A reusable simulation scenario. Launching produces N concrete `simulation_runs` rows.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| `id` | UUID | PK | |
| `workspace_id` | UUID | NOT NULL, FK `workspaces(id)` ON DELETE CASCADE | Scenarios are workspace-scoped per Rule 47. |
| `name` | varchar(255) | NOT NULL | |
| `description` | text | NULL | |
| `agents_config` | JSONB | NOT NULL | List of agent FQN references + per-agent overrides. Schema validated at the service layer. |
| `workflow_template_id` | UUID | NULL, FK `workflow_definitions(id)` ON DELETE RESTRICT | NULL for ad-hoc scenarios with inline workflow. |
| `mock_set_config` | JSONB | NOT NULL | Tool-mock declarations. The `MockLLMProvider` (UPD-044) is the default for LLM tools. |
| `input_distribution` | JSONB | NOT NULL | Per-input parameter distribution (uniform, normal, categorical, fixed). |
| `twin_fidelity` | JSONB | NOT NULL | Map of subsystem → `mock`/`real`. Validated against the existing `simulation_isolation_policies` for forbidden combinations. |
| `success_criteria` | JSONB | NOT NULL | Assertions against run outputs (latency, cost, content match). |
| `run_schedule` | JSONB | NULL | Optional cron-like schedule for automatic launches. NULL = manual launch only. |
| `archived_at` | timestamptz | NULL | Soft-archive; archived scenarios hidden from default lists. |
| `created_by` | UUID | NOT NULL, FK `users(id)` ON DELETE SET NULL | Creator. |
| `created_at` | timestamptz | NOT NULL, default `now()` | |
| `updated_at` | timestamptz | NOT NULL, default `now()`, ON UPDATE | |

**Indexes**:
- `IX_simulation_scenarios_workspace_id_archived_at` for default list (filter archived NULL).
- UNIQUE on `(workspace_id, name)` partial WHERE `archived_at IS NULL` — prevents duplicate active names.

**Validation rules** (enforced in `scenarios_service.py`):
1. No plaintext secrets in `mock_set_config` or `input_distribution` (regex scan against the standard secret patterns from `security_compliance/`).
2. Every agent FQN in `agents_config` resolves to an existing approved revision in the registry.
3. `workflow_template_id`, if set, must be approved for the workspace.
4. `twin_fidelity` cannot mix `real:production-data` with `mock:tool-gateway` (forbidden combination).
5. `success_criteria` must be a non-empty array.
