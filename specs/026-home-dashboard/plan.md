# Implementation Plan: Home Dashboard

**Branch**: `026-home-dashboard` | **Date**: 2026-04-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/026-home-dashboard/spec.md`

## Summary

Build the home dashboard page at `apps/web/app/(main)/home/page.tsx` within the Next.js 14+ App Router. The dashboard assembles four independently testable sections: a workspace summary MetricCard grid (4 metrics with change indicators), a recent activity Timeline feed (top 10 entries), a pending actions Card list (approvals, failed executions, attention requests sorted by urgency with inline approve/reject), and a Quick Actions button bar (4 navigation shortcuts with RBAC-based disabling). Real-time updates arrive via existing WebSocket channels (`execution`, `interaction`, `workspace`) and trigger TanStack Query invalidations. A `ConnectionStatusBanner` provides fallback status when the WebSocket connection is lost. All sections have independent error boundaries for partial failure isolation (FR-013).

## Key Decisions (from research.md)

1. **RSC shell + client root**: `page.tsx` is RSC; `HomeDashboard.tsx` is `"use client"` — supports future server-side prefetch while allowing WebSocket subscriptions
2. **TanStack Query staleTime 30s**: Matches WebSocket fallback polling interval; avoids excessive refetching when WebSocket is connected
3. **Query invalidation on WebSocket events**: Simpler and more reliable than manual query data patching; avoids response shape mismatches
4. **BFF aggregation endpoints**: Two new endpoints (`/dashboard/recent-activity`, `/dashboard/pending-actions`) avoid waterfall fetching and frontend-side merging/sorting — backend delivers exactly what the UI needs
5. **Optimistic removal of pending actions**: Approve/reject cards disappear immediately on click; reverted on error — prevents "stale card" UX
6. **Per-section error boundaries**: Each section fails independently (FR-013); one broken API does not white-screen the dashboard
7. **QUICK_ACTIONS static config**: No API call for quick actions; permission check uses existing auth store
8. **ConnectionStatusBanner placement**: Below page header, above content; `transition-all` for smooth appear/disappear
9. **Existing shared components**: `Timeline`, `MetricCard`, `StatusBadge`, `EmptyState` from feature 015 — no new primitive components
10. **Tailwind responsive breakpoints**: `sm:` (640px), `lg:` (1024px) — matches existing scaffold breakpoints
11. **MSW for offline development**: MSW handlers in `mocks/handlers/home.ts` allow full dashboard development without running backend
12. **Vitest + RTL + Playwright**: Inherited from feature 015 — no new tooling required
