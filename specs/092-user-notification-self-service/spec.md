## User Scenarios & Testing *(mandatory)*

### User Story 1 - User Reviews Missed Notifications (Priority: P1)

A user was offline yesterday or over the weekend; they were busy with operations during a high-traffic period (e.g., production execution storm + multiple budget alerts + an incident that resolved in their absence). Returning to the platform, they want a single place to catch up on what happened: `/notifications` paginated inbox + filter by date / channel / severity / event type / read state + bulk "mark all read". The notification bell in the global shell shows the unread count in real-time (driven by the existing `ALERTS` WebSocket channel per spec correction §5); clicking the bell opens a dropdown of the 5 most recent + a "See all" link. Drilling into a notification navigates to the originating resource (execution, goal, incident).

**Why this priority**: Constitutional Rule 45 ("Every user-facing backend capability has UI") makes this P1. Without a notification inbox UI, the existing `UserAlert` persistent backend (verified per research §6) is invisible to users — they receive emails but can never see the in-app history. P1 because (a) every other user-facing feature in the v1.3.0 audit-pass cohort assumes the notification bell exists; (b) the existing `<NotificationBell/>` placeholder in `Header.tsx:57` (per spec correction §6) is currently a stub that needs full implementation; (c) the existing backend endpoints (`/me/alerts`, `/me/alerts/unread-count`) work today and just need a UI consumer.

**Independent Test**: A test user is offline; the platform generates 15 `UserAlert` rows for them (6 execution-failed + 4 budget-alert + 3 goal-completed + 2 incident-resolved). User logs in; verifies (a) the bell badge shows `15` within 3 seconds of login (driven by the existing `GET /me/alerts/unread-count` endpoint); (b) clicking the bell opens a dropdown rendering the 5 most recent (newest first) via the existing `GET /me/alerts?limit=5&order=desc` endpoint; (c) clicking "See all" navigates to `/notifications`; (d) the inbox renders all 15 with cursor-based pagination via the existing `GET /me/alerts` endpoint; (e) filtering to "last 24h" + "severity=critical" narrows results server-side (the existing endpoint supports filter params per research §1); (f) clicking "Mark all as read" calls the NEW `POST /me/alerts/mark-all-read` endpoint (added in this feature); the bell badge clears to `0` within 2 seconds; (g) clicking an individual notification (e.g., an execution-failed alert) navigates to `/executions/{execution_id}` (the originating resource — derived from `UserAlert.source_reference` JSONB column verified per research §6); (h) notifications older than 90 days (the FR-649 retention window) are not present in the inbox (archived by the existing retention scheduler — verify path during plan phase).

**Acceptance Scenarios**:

1. **Given** a user with 15 unread `UserAlert` rows, **When** they log in, **Then** the bell badge shows 15 within 3 seconds AND the bell renders as a sticky element in the global header.
2. **Given** the user clicks the bell, **When** the dropdown opens, **Then** the 5 most recent alerts render with title + relative timestamp + severity icon.
3. **Given** the user navigates to `/notifications`, **When** they apply filters (date range, severity, channel, event type, read state), **Then** results narrow server-side (no client-side filtering — the existing endpoint already supports filter params per research §1).
4. **Given** the user clicks "Mark all as read", **When** the bulk action completes, **Then** all 15 alerts have `read=true` AND the bell badge clears to 0 within 2 seconds.
5. **Given** the user clicks an individual alert with `source_reference={"type": "execution", "id": "..."}`, **When** the click is processed, **Then** the user is navigated to `/executions/{id}` AND the alert is marked read on view (existing `PATCH /me/alerts/{id}/read`).
6. **Given** a high-volume notification storm (10,000 alerts in an hour per spec edge case), **When** the user views the inbox, **Then** pagination kicks in (default page size 50) AND a banner offers to enable digest mode for the offending event type.

### User Story 2 - User Configures Notification Preferences (Priority: P1)

A user wants email only for critical events (execution-failed, incident-resolved), Slack for everyday updates (goal-completed, budget-warning), and explicitly does NOT want SMS. Quiet hours: no notifications between 22:00 and 07:00 in their timezone. Mandatory events (security: login-from-new-location, MFA-disabled, password-changed) cannot be fully disabled per the spec — these are forced-on with a tooltip explaining why.

**Why this priority**: Per FR-651 + Rule 45, the preferences matrix is P1 because (a) without it, every user receives the same notifications via the same channel (the existing `UserAlertSettings` schema is insufficient per spec correction §7); (b) the brownfield's claim "the backend is proven" is INCORRECT for preferences specifically — the existing schema needs extension; (c) without the matrix, multi-channel delivery (UPD-077, already shipped per spec correction §1) is wasted because users can't tune which event goes where.

**Independent Test**: A user navigates to `/settings/notifications` (NEW page). Configures: email = critical-only (execution-failed, incident-resolved); slack = everyday (goal-completed, budget-warning); SMS = disabled; quiet hours = 22:00-07:00 user-local timezone. Saves. Verify (a) the matrix UI renders an N-event × 6-channel grid (the 6 channels per spec correction §1); (b) on save, the new schema (per spec correction §7 — extension of `UserAlertSettings` OR new sibling table) persists the preferences; (c) generates synthetic events at 14:00 (an execution-failed) → user receives email + slack (both selected for critical), no SMS; (d) generates a goal-completed at 14:30 → user receives slack only; (e) generates an MFA-disabled event at 14:35 → user receives ALL configured channels (mandatory event, can't be disabled); (f) generates a goal-completed at 22:30 (quiet hours) → no notification delivered until 07:00 next day; (g) clicks "Send test notification" for execution-failed → triggers a real synthetic delivery; verify the email lands within 30 seconds.

**Acceptance Scenarios**:

1. **Given** the matrix UI loaded with the user's existing preferences, **When** they toggle a cell (event × channel), **Then** the change is queued client-side and saved on Save click.
2. **Given** the user attempts to disable a mandatory event (e.g., `security.mfa_disabled`), **When** they click the toggle, **Then** the toggle is locked AND a tooltip explains "This event is mandatory; it cannot be disabled".
3. **Given** quiet hours are 22:00-07:00 user-local, **When** an event arrives at 22:30, **Then** the delivery is queued for 07:00 the next day (digest mode auto-applied) — UNLESS the event is critical (incident-fired, security.*) in which case it bypasses quiet hours.
4. **Given** the user clicks "Send test notification" for an event type, **When** the action fires, **Then** a synthetic alert is generated AND delivered via the user's configured channels for that event type.
5. **Given** preference changes are saved, **When** the next real event of that type fires, **Then** routing reflects the new preferences (no caching delay beyond the platform's existing 30-second cache TTL).

### User Story 3 - Developer Manages Their Personal API Tokens (Priority: P1)

A developer building a custom integration with the platform needs a long-lived API token scoped to read-only access for their workspace. They navigate to `/settings/api-keys` (NEW page), click "Create new key", enter a name + scope subset + expiry, click Confirm, copy the displayed token value (one-time render per Rule 31), and use it to call `/api/v1/me`. Later, when the token is no longer needed, they revoke it from the same page.

**Why this priority**: Per FR-652 + Rule 45, P1 because (a) the existing `POST /api/v1/auth/service-accounts` endpoint is platform-admin-only per spec correction §3 — users CANNOT create their own tokens today; (b) every external integrator needs this capability; (c) the existing `ServiceAccountCredential` model (verified per research §8) is reused unchanged with a NEW `created_by_user_id` column to distinguish self-service tokens from admin-created ones.

**Independent Test**: A user with MFA enabled navigates to `/settings/api-keys`. Clicks "Create new API key"; modal opens with name input, scope multi-select (drawn from the user's own permission set — the user CANNOT request scope they don't have themselves per spec edge case + FR-652), expiry dropdown (default 90 days, max 1 year configurable per platform setting). Submits; the modal triggers an MFA step-up challenge (per FR-652 + spec User Story 4 acceptance scenario 5 — MFA is enabled). Enter TOTP; the backend creates the row with `created_by_user_id=current_user.id`, returns the raw token value ONCE (per Rule 31 + spec User Story 3 acceptance scenario 3). The UI displays the token in a `<CodeBlock>` with prominent copy + warning copy "This is the only time you'll see this value". User copies, dismisses modal. Verify (a) the list view renders the new key with name, scope, created-at, last-used (initially null), expires-at, revoke button; (b) attempting to view the value again is impossible (no endpoint returns the raw value); (c) using the token via `curl -H "Authorization: Bearer ${TOKEN}" /api/v1/me` returns the user's profile; (d) clicks "Revoke"; backend sets `status=revoked`; immediate-effect verified by next `curl` returning 401; (e) the user has 11 active tokens — attempts to create a 12th — UI rejects with "Maximum 10 personal API keys" per spec correction §3.

**Acceptance Scenarios**:

1. **Given** a user with MFA enabled, **When** they create an API key, **Then** the action requires MFA step-up; on success the token value is displayed once + the row is created with `created_by_user_id=current_user.id`.
2. **Given** the user dismisses the create modal, **When** they later view the list, **Then** the token value is NOT retrievable from any endpoint (only the metadata: name, scope, last_used_at, expires_at).
3. **Given** the user attempts to request a scope they don't have themselves (e.g., a member requesting `admin.*` scope), **When** the create-API-key request hits the backend, **Then** the request is rejected with HTTP 403 + a clear UI rejection.
4. **Given** the user's token count is at the max (10 per user), **When** they attempt to create an 11th, **Then** the create button is disabled + a tooltip explains "Maximum 10 personal API keys; revoke an existing one first".
5. **Given** the user clicks "Revoke" on a key, **When** the action fires, **Then** the backend marks `status=revoked` AND the next API call using that token returns 401 within 5 seconds.

### Edge Cases

- **Notification volume spike (10,000 alerts in an hour).** The existing `UserAlert` table can hold the volume; the inbox UI defaults to page size 50 + offers a banner "We detected high notification volume; enable digest mode for [event_type]?" — clicking the banner auto-toggles digest mode for that event type in `/settings/notifications`.
- **User attempts to revoke their CURRENT session.** The backend `DELETE /me/sessions/{id}` endpoint refuses with HTTP 400 + error message "You can't revoke the session you're currently using"; the UI's "Revoke" button is disabled on the current-session row.
- **User disables MFA when admin policy enforces it.** The existing MFA enforcement policy (added in this feature per spec correction §13) raises 403 on the disable endpoint with a clear error referring to admin policy; the UI's "Disable MFA" button is disabled with a tooltip "MFA is enforced by your administrator and cannot be disabled".
- **API key creation with insufficient scope privileges.** The user requests scope `admin.*` they don't have; backend rejects with 403; UI surfaces "You can't create an API key with permissions you don't have yourself".
- **DSR erasure with active executions.** The submission UI surfaces a warning panel listing the active executions; the user must check a box "I understand pending executions will fail" before submitting.
- **Concurrent consent revocation and new interaction.** A user clicks "Revoke ai_interaction" while they have an in-flight conversation; the backend blocks new interactions immediately (next request fails) but allows the in-flight one to complete (no mid-stream interruption); the user sees the consent-revoked banner on next interaction attempt.
- **MFA backup codes lost.** User has lost both phone + backup codes; UI directs to the password-recovery flow + admin-assisted MFA reset (existing UPD-016 / UPD-036 admin endpoint — exact path verified during plan phase).
- **Notification preference change during high-load.** User changes preferences while a 10,000-event burst is mid-flight; the new preferences apply only to events arriving AFTER the change is persisted (the existing 30-second cache TTL applies); in-flight events use the prior preferences.
- **DSR completion email lost.** The download link emailed on DSR completion has a 7-day TTL; if the user misses it, they can re-request via the existing admin-DSR `retry` endpoint (per research §11) — the user MAY contact support to trigger; a "Re-send download link" button on the user's DSR detail page is a future enhancement.
- **Policy version change during active session.** The user is logged in when an admin updates a consent policy version; the user's existing consent for the prior version remains valid (no automatic revocation); the next login OR an explicit refresh prompt asks them to re-consent.

  - **Admin-on-behalf-of-user impersonation flow improvements.** The existing UPD-024 / UPD-036 impersonation paths are reused unchanged; the Rule 34 double-audit applies per spec correction §10.
  - **DSR re-send download link button.** A future enhancement; today the user contacts support OR an admin runs `POST /admin/dsr/{id}/retry`.
  - **MFA backup-codes recovery flow.** UPD-042 ships the regenerate path (requires current TOTP step-up); a "lost both phone + backup codes" recovery flow is admin-assisted via the existing UPD-016 / UPD-036 admin endpoint.
  - **Per-workspace notification preferences.** UPD-042 ships per-USER preferences only; per-workspace overrides (e.g., a user wants different preferences in workspace A vs B) is a future enhancement.
  - **Real-time `mark-as-read` cross-tab sync via the WS channel.** SC-003 mentions this but the implementation is deferred — the cross-tab sync is OPTIONAL and may be added in a follow-up; the MVP is single-tab-per-session correct.
  - **Notification archive / trash / unarchive.** The 90-day retention is hard-deletion (per FR-649); a soft-delete archive is a future enhancement.
  - **Per-event-type custom delivery rules** (e.g., "deliver execution-failed events for workspace X via slack to channel #ops"). The matrix is per-event × channel; per-workspace override is a future enhancement.
  - **Notification grouping / threading.** Each `UserAlert` is a discrete row; conversation-style grouping is a future enhancement.
  - **Email digest format customization.** The digest mode (hourly / daily) ships with a default email template; user-customizable HTML / preference is a future enhancement.
  - **Removing the existing platform-admin `POST /api/v1/auth/service-accounts` endpoint.** Per spec correction §3 — the existing platform-admin endpoint is PRESERVED for system-level service accounts; UPD-042 ADDS `/me/service-accounts` for user-self.
